# 4.3 MongoDB jako warstwa danych: Baza danych, Auth i Vector Search w stacku z Node.js

**Moduł 4: Szybkie Wdrażanie (Ship Fast) w oparciu o Angular**

---

## Teoria

### 1. MongoDB w stacku MEAN — dlaczego pasuje do aplikacji AI?

MongoDB to **dokumentowa baza danych NoSQL**, przechowująca dane w formacie BSON (binarny JSON). W kontekście aplikacji AI jej elastyczność jest wyjątkową zaletą: dokumenty nie muszą mieć stałego schematu, co ułatwia iterowanie nad strukturą danych podczas szybkiego prototypowania.

Dlaczego MongoDB dobrze pasuje do stacku AI:

| Cecha | Korzyść dla aplikacji AI |
|-------|--------------------------|
| **Schemat elastyczny** | Historia czatu, wyniki AI i metadane mają różną strukturę — MongoDB nie wymaga migracji |
| **Zagnieżdżone dokumenty** | Wiadomości w sesji czatu to naturalnie tablica obiektów w jednym dokumencie |
| **Atlas Vector Search** | Wbudowane wyszukiwanie semantyczne bez osobnego serwisu (omówione w sekcji 3.2) |
| **Atlas Search** | Pełnotekstowe przeszukiwanie — przydatne w historii czatu lub dokumentach |
| **Mongoose (ODM)** | Warstwa walidacji i schematów po stronie Node.js — opcjonalna, ale wygodna |

Połączenie z Node.js realizujemy przez oficjalny sterownik `mongodb` lub przez **Mongoose** — Object Document Mapper dodający schematy, walidację i middleware (hooks).

### 2. Połączenie z MongoDB w Node.js i zarządzanie połączeniem

Dobra praktyka: połączenie z bazą tworzymy **raz przy starcie aplikacji** i wielokrotnie je reużywamy — nie otwieramy nowego połączenia przy każdym żądaniu HTTP.

**Instalacja:**

```bash
npm install mongodb
# lub z Mongoose:
npm install mongoose
```

**Singleton połączenia z natywnym sterownikiem:**

```javascript
// db.js
import { MongoClient } from 'mongodb';

let client;
let db;

export async function connectDB() {
  if (db) return db;  // reużyj istniejącego połączenia

  client = new MongoClient(process.env.MONGODB_URI);
  await client.connect();
  db = client.db(process.env.DB_NAME ?? 'aiapp');
  console.log('Połączono z MongoDB');
  return db;
}

export function getDB() {
  if (!db) throw new Error('Baza danych nie jest zainicjalizowana. Wywołaj connectDB() najpierw.');
  return db;
}
```

**W aplikacji Express:**

```javascript
// server.js
import express from 'express';
import { connectDB } from './db.js';

const app = express();
app.use(express.json());

// Importuj routery po inicjalizacji
import chatRouter from './routes/chat.js';
app.use('/api/chat', chatRouter);

await connectDB();
app.listen(3000, () => console.log('Serwer na porcie 3000'));
```

Zmienna `MONGODB_URI` przechowuje connection string w formacie:
```
mongodb+srv://user:password@cluster.mongodb.net/?retryWrites=true
```
Nigdy nie umieszczaj connection string bezpośrednio w kodzie — zawsze używaj zmiennych środowiskowych i pliku `.env` (lokalnie) lub AWS Secrets Manager (produkcja).

### 3. Modelowanie danych dla aplikacji AI — sesje i wiadomości

W aplikacji czatu AI typowa struktura danych to **sesja** (session) zawierająca historię wiadomości. Dwa podejścia projektowe:

**Podejście A — wiadomości zagnieżdżone w sesji (embedded):**
```javascript
// Dokument sesji czatu
{
  _id: ObjectId("..."),
  userId: "user_123",
  title: "Pytania o MongoDB",
  createdAt: ISODate("2026-04-16T10:00:00Z"),
  messages: [
    { role: "user",      content: "Czym jest indeks?",       ts: ISODate("...") },
    { role: "assistant", content: "Indeks to struktura...",   ts: ISODate("...") }
  ]
}
```
Zalety: jeden odczyt = cała sesja. Wada: dokument może przekroczyć limit 16 MB przy bardzo długich sesjach.

**Podejście B — wiadomości w osobnej kolekcji (referenced):**
```javascript
// Kolekcja sessions
{ _id: ObjectId("s1"), userId: "user_123", title: "...", createdAt: ISODate("...") }

// Kolekcja messages
{ _id: ObjectId("m1"), sessionId: ObjectId("s1"), role: "user", content: "...", ts: ISODate("...") }
{ _id: ObjectId("m2"), sessionId: ObjectId("s1"), role: "assistant", content: "...", ts: ISODate("...") }
```
Zalety: brak limitu rozmiaru, łatwiejsze stronicowanie wiadomości. Wada: dwa zapytania zamiast jednego.

Dla typowych aplikacji AI (sesje do kilkuset wiadomości) **podejście A** jest prostsze i wystarczające.

### 4. Autoryzacja z JWT w stacku Node.js + MongoDB

W aplikacjach AI ważne jest, by każdy użytkownik widział tylko swoje sesje i historię. Standardowym podejściem jest **JWT (JSON Web Token)** — bezstanowy token podpisany kluczem tajnym.

**Przepływ autoryzacji:**

```
[Angular] → POST /api/auth/login { email, password }
    → [Node.js] sprawdź hasło (bcrypt) w MongoDB
    → wygeneruj JWT (payload: userId, exp)
    → zwróć token do Angular
[Angular] → przechowaj token (localStorage lub httpOnly cookie)
[Angular] → każde żądanie: Authorization: Bearer <token>
    → [Node.js middleware] weryfikuje podpis JWT
    → dodaje userId do req
    → kontroler filtruje dane po userId
```

**Implementacja middleware autoryzującego:**

```javascript
// middleware/auth.js
import jwt from 'jsonwebtoken';

export function requireAuth(req, res, next) {
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'Brak tokenu autoryzacyjnego' });
  }

  const token = authHeader.slice(7);
  try {
    const payload = jwt.verify(token, process.env.JWT_SECRET);
    req.userId = payload.userId;
    next();
  } catch {
    res.status(401).json({ error: 'Nieprawidłowy lub wygasły token' });
  }
}
```

```javascript
// routes/chat.js — chronione trasy
import { requireAuth } from '../middleware/auth.js';

router.get('/sessions', requireAuth, async (req, res) => {
  const db = getDB();
  const sessions = await db.collection('sessions')
    .find({ userId: req.userId })         // tylko sesje zalogowanego użytkownika
    .sort({ createdAt: -1 })
    .toArray();
  res.json(sessions);
});
```

**Ważne zasady bezpieczeństwa:**
- Przechowuj `JWT_SECRET` wyłącznie w zmiennych środowiskowych (nie w kodzie).
- Ustawiaj krótki czas wygaśnięcia tokenu (`exp: '1h'`) i implementuj refresh tokens dla dłuższych sesji.
- Używaj `bcrypt` do hashowania haseł (nigdy czystego tekstu w bazie).

### 5. Indeksy MongoDB — wydajność zapytań

Bez indeksów MongoDB wykonuje pełne skanowanie kolekcji (ang. *collection scan*) przy każdym zapytaniu. Gdy kolekcja rośnie do tysięcy dokumentów, czas odpowiedzi dramatycznie wzrasta.

**Tworzenie indeksów w Node.js:**

```javascript
async function ensureIndexes(db) {
  const sessions = db.collection('sessions');
  const messages = db.collection('messages');

  // Indeks dla zapytań po userId (filtrowanie sesji użytkownika)
  await sessions.createIndex({ userId: 1, createdAt: -1 });

  // Indeks TTL — automatyczne usuwanie sesji po 90 dniach
  await sessions.createIndex(
    { createdAt: 1 },
    { expireAfterSeconds: 90 * 24 * 60 * 60 }
  );

  // Indeks dla paginacji wiadomości w sesji
  await messages.createIndex({ sessionId: 1, ts: 1 });
}
```

**Typy indeksów MongoDB przydatnych w aplikacjach AI:**

| Typ | Zastosowanie |
|-----|-------------|
| **Single field** | Filtrowanie po `userId`, `sessionId` |
| **Compound** | Sortowanie po wielu polach jednocześnie |
| **TTL** | Automatyczne wygasanie danych tymczasowych |
| **Text** | Pełnotekstowe przeszukiwanie historii czatu |
| **Vector** | Semantyczne wyszukiwanie embeddingów (Atlas Vector Search) |

Wywołaj `ensureIndexes()` przy starcie aplikacji — MongoDB pomija tworzenie, jeśli indeks już istnieje.

### 6. Środowisko deweloperskie — MongoDB w Dockerze

Podczas developmentu uruchamiamy MongoDB lokalnie przez Docker, by nie połączać się z klastrem Atlas (oszczędność kosztów, brak zależności od sieci):

```yaml
# docker-compose.yml (development)
services:
  mongo:
    image: mongo:7
    ports:
      - "27017:27017"
    volumes:
      - mongo_data:/data/db
    environment:
      MONGO_INITDB_DATABASE: aiapp

  mongo-express:
    image: mongo-express:latest
    ports:
      - "8081:8081"
    environment:
      ME_CONFIG_MONGODB_SERVER: mongo
      ME_CONFIG_BASICAUTH: "false"
    depends_on:
      - mongo

volumes:
  mongo_data:
```

```bash
docker compose up -d mongo mongo-express
# MongoDB dostępne na localhost:27017
# Mongo Express (GUI) na http://localhost:8081
```

**Zmienne środowiskowe dla różnych środowisk:**

```bash
# .env (development — Docker lokalny)
MONGODB_URI=mongodb://localhost:27017
DB_NAME=aiapp
JWT_SECRET=dev-secret-change-in-prod

# .env.production (Atlas — produkcja)
MONGODB_URI=mongodb+srv://user:pass@cluster.mongodb.net/
DB_NAME=aiapp_prod
JWT_SECRET=<wartość z AWS Secrets Manager>
```

W produkcji (AWS Lambda) zmienne środowiskowe definiujemy bezpośrednio w konfiguracji funkcji Lambda lub przez AWS Secrets Manager — nigdy nie umieszczamy ich w repozytorium GitHub.

## Zadania praktyczne

### Zadanie 1 — Łatwe
**Lokalna baza danych MongoDB w Dockerze**

Uruchom MongoDB i Mongo Express za pomocą pliku `docker-compose.yml`. Napisz skrypt Node.js (`seed.js`), który połączy się z lokalną bazą i wstawi do kolekcji `sessions` trzy przykładowe dokumenty z polami `userId`, `title` i `createdAt`. Zweryfikuj poprawność zapisu w interfejsie Mongo Express pod adresem `http://localhost:8081`.

---

### Zadanie 2 — Średnie
**Endpoint REST dla sesji czatu z autoryzacją JWT**

Zaimplementuj w Node.js/Express dwie trasy:
- `POST /api/auth/login` — przyjmuje `{ email, password }`, weryfikuje dane z kolekcji `users` (hasło hashowane `bcrypt`) i zwraca JWT,
- `GET /api/sessions` — chroniona przez middleware `requireAuth`, zwraca tylko sesje zalogowanego użytkownika (`userId` z tokenu JWT).

Przetestuj całość za pomocą `curl` lub Postmana: najpierw zaloguj się i pobierz token, następnie użyj go do pobrania sesji.

---

### Zadanie 3 — Średnie
**Modelowanie i stronicowanie historii wiadomości**

Zaimplementuj kolekcję `messages` z dokumentami zawierającymi pola `sessionId`, `role`, `content` i `ts`. Utwórz odpowiednie indeksy (`sessionId + ts`). Następnie napisz endpoint `GET /api/sessions/:id/messages?page=1&limit=20`, który zwraca stronicowaną historię wiadomości dla danej sesji (używając `skip` i `limit` lub kursora MongoDB). Zweryfikuj wydajność zapytań, uruchamiając `explain("executionStats")` i sprawdzając, czy zapytanie korzysta z indeksu.

---

### Zadanie 4 — Zaawansowane
**Pełny stack Angular + Node.js + MongoDB z autoryzacją**

Zbuduj kompletną aplikację end-to-end:
1. Ekran logowania w Angular (Angular Material `mat-form-field`) wysyłający dane do `POST /api/auth/login` i zapisujący JWT w `localStorage`.
2. Interceptor Angular HTTP (`HttpInterceptor`) automatycznie dodający nagłówek `Authorization: Bearer <token>` do każdego żądania.
3. Widok listy sesji (`GET /api/sessions`) i widok szczegółów sesji z historią wiadomości (`GET /api/sessions/:id/messages`) — z obsługą stronicowania.
4. Guard Angular (`CanActivate`) przekierowujący niezalogowanych użytkowników na stronę logowania.
5. Całość uruchom w środowisku Docker Compose (Angular dev server, Node.js, MongoDB). Przetestuj scenariusz pełnego przepływu: rejestracja → logowanie → tworzenie sesji → historia wiadomości.
