# 5.3 Self-healing Infrastructure: Budowa agentów monitorujących logi i automatycznie sugerujących poprawki w infrastrukturze AWS

**Moduł 5: Agenty AI w procesach CI/CD i DevOps**

---

## Teoria

### 1. Czym jest self-healing infrastructure?

**Self-healing infrastructure** (samonaprawiająca się infrastruktura) to podejście, w którym system automatycznie wykrywa anomalie, awarie lub degradację wydajności, a następnie podejmuje działania naprawcze — bez interwencji człowieka lub informując go o gotowej sugestii poprawki.

Tradycyjny model reagowania na incydenty wygląda następująco:
1. System generuje alert.
2. Inżynier odczytuje alert (być może w środku nocy).
3. Inżynier analizuje logi i wdraża poprawkę.

W modelu self-healing agent AI przejmuje kroki 2 i 3 — analizuje logi, identyfikuje przyczynę i albo **automatycznie stosuje poprawkę** (dla przewidywalnych, bezpiecznych scenariuszy), albo **generuje gotową do zatwierdzenia sugestię** (dla zmian wymagających nadzoru ludzkiego).

W kontekście stacku AWS kluczowe usługi biorące udział w tym procesie:

| Usługa AWS | Rola w self-healing |
|------------|---------------------|
| **CloudWatch Logs** | Źródło danych — logi aplikacji i infrastruktury |
| **CloudWatch Alarms** | Wyzwalacz — alert przy przekroczeniu progu metryki |
| **SNS (Simple Notification Service)** | Kanał — dostarcza powiadomienie do Lambda lub email |
| **Lambda** | Wykonawca — uruchamia agenta AS w odpowiedzi na alarm |
| **AWS SDK** | Narzędzie — agent wykonuje akcje naprawcze (restart, skalowanie) |
| **GitHub API** | Kanał sugestii — agent otwiera PR z poprawką konfiguracji |

### 2. Architektura agenta self-healing — przepływ zdarzeń

Typowy przepływ agenta monitorującego logi AWS Lambda i sugerującego poprawki:

```
[AWS CloudWatch Logs]
        │  logi Lambda (błędy, timeouty, cold starty)
        ▼
[CloudWatch Metric Filter]
        │  filtruje zdarzenia: np. "Task timed out", "Error:"
        ▼
[CloudWatch Alarm]
        │  próg: >5 błędów w ciągu 5 minut
        ▼
[SNS Topic]
        │  wiadomość JSON z informacją o alarmie
        ▼
[Lambda — agent self-healing]
        │
        ├─ 1. Pobierz ostatnie logi (CloudWatch Logs API)
        ├─ 2. Wyślij logi do LLM (OpenAI) z prośbą o diagnozę
        ├─ 3a. Bezpieczna poprawka → wykonaj natychmiast (np. zwiększ timeout)
        └─ 3b. Złożona poprawka → otwórz PR na GitHubie z sugestią
```

Podział na **działania automatyczne** (3a) i **sugestie do zatwierdzenia** (3b) jest kluczowy bezpieczeństwo — agent nie powinien mieć nieograniczonych uprawnień do modyfikowania infrastruktury produkcyjnej.

### 3. Pobieranie i analiza logów CloudWatch przez Node.js

Agent potrzebuje dostępu do niedawnych logów aplikacji. AWS SDK dla JavaScript v3 udostępnia klienta `CloudWatchLogsClient`.

```javascript
// agent/logs.js
import {
  CloudWatchLogsClient,
  FilterLogEventsCommand,
} from '@aws-sdk/client-cloudwatch-logs';

const logsClient = new CloudWatchLogsClient({ region: 'eu-west-1' });

export async function getRecentErrors(logGroupName, minutesBack = 30) {
  const startTime = Date.now() - minutesBack * 60 * 1000;

  const response = await logsClient.send(new FilterLogEventsCommand({
    logGroupName,
    startTime,
    filterPattern: '?ERROR ?Error ?error ?"Task timed out"',
    limit: 50,
  }));

  return response.events?.map(e => ({
    timestamp: new Date(e.timestamp).toISOString(),
    message: e.message.trim(),
  })) ?? [];
}
```

```javascript
// Użycie w agencie
const errors = await getRecentErrors('/aws/lambda/ai-app-backend');

if (errors.length === 0) {
  console.log('Brak błędów w logach.');
  return;
}

const logSummary = errors
  .map(e => `[${e.timestamp}] ${e.message}`)
  .join('\n');
```

Filtr `?ERROR ?Error ?error ?"Task timed out"` to składnia CloudWatch Logs — znak `?` oznacza „zawiera którekolwiek z tych słów". Zwróć uwagę na `limit: 50` — nie pobieraj nieograniczonej liczby logów (koszty API i tokeny LLM).

### 4. Diagnoza i sugestia poprawki przez LLM

Rdzeń agenta to wysłanie zebranych logów do LLM z odpowiednio skonstruowanym prompem. Prompt musi zawierać kontekst infrastruktury, żeby LLM mógł zaproponować konkretne, wykonalne poprawki.

```javascript
// agent/diagnose.js
import OpenAI from 'openai';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

export async function diagnoseAndSuggest(logSummary, functionConfig) {
  const response = await openai.chat.completions.create({
    model: 'gpt-4o',
    messages: [
      {
        role: 'system',
        content: `Jesteś ekspertem AWS DevOps. Analizujesz logi AWS Lambda i sugerujesz poprawki infrastruktury.
Stack: Node.js 22, MongoDB Atlas, AWS Lambda, API Gateway.

Dla każdego problemu określ:
1. Przyczynę (root cause)
2. Kategorię: "config" (zmiana konfiguracji Lambda), "code" (błąd w kodzie), "external" (problem zewnętrzny — MongoDB, OpenAI)
3. Dla kategorii "config" — konkretną zmianę konfiguracji Lambda (timeout, memory, env var)
4. Dla kategorii "code" — opis błędu do naprawy w kodzie

Odpowiedz w formacie JSON:
{
  "summary": "krótkie podsumowanie problemu",
  "rootCause": "opis przyczyny",
  "category": "config|code|external",
  "autoFix": { "type": "timeout|memory|env", "value": "..." } | null,
  "codeSuggestion": "opis poprawki kodu" | null
}`,
      },
      {
        role: 'user',
        content: `Konfiguracja funkcji Lambda:
- Timeout: ${functionConfig.timeout}s
- Pamięć: ${functionConfig.memorySize}MB
- Runtime: ${functionConfig.runtime}

Logi błędów (ostatnie 30 minut):
\`\`\`
${logSummary}
\`\`\`

Zdiagnozuj problem i zaproponuj poprawkę.`,
      },
    ],
    response_format: { type: 'json_object' },
  });

  return JSON.parse(response.choices[0].message.content);
}
```

### 5. Automatyczne stosowanie bezpiecznych poprawek konfiguracyjnych

Gdy LLM wskaże kategorię `config` z konkretną wartością (np. zwiększenie timeoutu), agent może automatycznie zastosować zmianę przez AWS SDK — bez otwierania PR czy angażowania człowieka.

```javascript
// agent/fix.js
import { LambdaClient, UpdateFunctionConfigurationCommand } from '@aws-sdk/client-lambda';

const lambdaClient = new LambdaClient({ region: 'eu-west-1' });

export async function applyConfigFix(functionName, autoFix) {
  const update = {};

  if (autoFix.type === 'timeout') {
    const newTimeout = Math.min(parseInt(autoFix.value), 900); // max 900s dla Lambda
    update.Timeout = newTimeout;
  } else if (autoFix.type === 'memory') {
    const newMemory = Math.min(parseInt(autoFix.value), 10240); // max 10240 MB
    update.MemorySize = newMemory;
  }

  if (Object.keys(update).length === 0) return null;

  await lambdaClient.send(new UpdateFunctionConfigurationCommand({
    FunctionName: functionName,
    ...update,
  }));

  console.log(`Zastosowano poprawkę konfiguracji: ${JSON.stringify(update)}`);
  return update;
}
```

**Zasada minimalnych uprawnień (Principle of Least Privilege):** rola IAM agenta powinna mieć dostęp wyłącznie do niezbędnych akcji:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "lambda:UpdateFunctionConfiguration",
        "lambda:GetFunctionConfiguration",
        "logs:FilterLogEvents",
        "logs:DescribeLogGroups"
      ],
      "Resource": "arn:aws:lambda:eu-west-1:123456789:function:ai-app-backend"
    }
  ]
}
```

Nigdy nie nadawaj agentowi uprawnienia `*` ani `lambda:*` — ograniczaj do konkretnych akcji i konkretnych zasobów ARN.

### 6. Sugestie poprawek kodu — automatyczny Pull Request na GitHubie

Gdy LLM zidentyfikuje błąd w kodzie (kategoria `code`), agent nie może samodzielnie naprawić aplikacji — powinien otworzyć Pull Request z opisem problemu i sugestią. To podejście łączy automatyzację z kontrolą człowieka.

```javascript
// agent/suggest-pr.js
import { Octokit } from '@octokit/rest';

const octokit = new Octokit({ auth: process.env.GITHUB_TOKEN });

export async function createSuggestionPR(owner, repo, diagnosis) {
  const timestamp = new Date().toISOString().replace(/[:.]/g, '-');
  const branchName = `fix/self-healing-${timestamp}`;

  // Pobierz SHA głównej gałęzi
  const { data: ref } = await octokit.git.getRef({
    owner, repo, ref: 'heads/main',
  });

  // Utwórz nową gałąź
  await octokit.git.createRef({
    owner, repo,
    ref: `refs/heads/${branchName}`,
    sha: ref.object.sha,
  });

  // Otwórz PR z opisem diagnozy
  const { data: pr } = await octokit.pulls.create({
    owner, repo,
    title: `[Self-healing] ${diagnosis.summary}`,
    head: branchName,
    base: 'main',
    body: `## 🤖 Automatyczna sugestia self-healing agenta

**Przyczyna błędu:** ${diagnosis.rootCause}

**Sugestia poprawki:**
${diagnosis.codeSuggestion}

---
*Sugestia wygenerowana automatycznie przez agenta monitorującego logi CloudWatch.*
*Przejrzyj i zatwierdź zmiany przed scaleniem.*`,
  });

  console.log(`Otworzono PR #${pr.number}: ${pr.html_url}`);
  return pr;
}
```

Taki PR trafia do kolejki przeglądu — deweloper widzi diagnozę i sugestię, może ją odrzucić lub zaimplementować. Bot z sekcji 5.2 może automatycznie przejrzeć kod zaproponowany w tym PR.

### 7. Wdrożenie agenta jako Lambda wyzwalanej przez SNS

Cały agent działa jako osobna funkcja Lambda subskrybująca temat SNS (powiązany z alarmem CloudWatch).

```javascript
// agent/handler.js — punkt wejścia Lambda
import { getRecentErrors } from './logs.js';
import { diagnoseAndSuggest } from './diagnose.js';
import { applyConfigFix } from './fix.js';
import { createSuggestionPR } from './suggest-pr.js';
import { LambdaClient, GetFunctionConfigurationCommand } from '@aws-sdk/client-lambda';

const lambdaClient = new LambdaClient({ region: 'eu-west-1' });
const TARGET_FUNCTION = process.env.MONITORED_FUNCTION_NAME;

export const handler = async (event) => {
  console.log('Agent self-healing uruchomiony:', JSON.stringify(event));

  // 1. Pobierz logi
  const errors = await getRecentErrors(`/aws/lambda/${TARGET_FUNCTION}`);
  if (errors.length === 0) return { status: 'no_errors' };

  const logSummary = errors.map(e => `[${e.timestamp}] ${e.message}`).join('\n');

  // 2. Pobierz konfigurację monitorowanej funkcji
  const { data: config } = await lambdaClient.send(
    new GetFunctionConfigurationCommand({ FunctionName: TARGET_FUNCTION })
  );
  const functionConfig = {
    timeout: config.Timeout,
    memorySize: config.MemorySize,
    runtime: config.Runtime,
  };

  // 3. Diagnoza przez LLM
  const diagnosis = await diagnoseAndSuggest(logSummary, functionConfig);
  console.log('Diagnoza:', JSON.stringify(diagnosis));

  // 4. Działanie zależne od kategorii
  if (diagnosis.category === 'config' && diagnosis.autoFix) {
    const applied = await applyConfigFix(TARGET_FUNCTION, diagnosis.autoFix);
    return { status: 'auto_fixed', applied };
  }

  if (diagnosis.category === 'code' && diagnosis.codeSuggestion) {
    const pr = await createSuggestionPR(
      process.env.GITHUB_OWNER,
      process.env.GITHUB_REPO,
      diagnosis
    );
    return { status: 'pr_created', prUrl: pr.html_url };
  }

  return { status: 'external_issue', summary: diagnosis.summary };
};
```

## Zadania praktyczne

### Zadanie 1 — Łatwe
**Ręczna analiza logów CloudWatch przez LLM**

W konsoli AWS CloudWatch Logs otwórz strumień logów istniejącej funkcji Lambda (może to być funkcja z sekcji 4.4). Skopiuj 10–20 ostatnich wpisów zawierających błędy lub ostrzeżenia. Wklej je do GitHub Copilot Chat lub ChatGPT z prośbą: „Zdiagnozuj problem w tych logach Lambda i zaproponuj konkretną poprawkę konfiguracji lub kodu." Porównaj sugestię AI z własną analizą i oceń jej trafność.

---

### Zadanie 2 — Średnie
**Agent logów uruchamiany ręcznie przez GitHub Actions**

Zaimplementuj skrypt Node.js `agent/diagnose-logs.js` pobierający ostatnie błędy z CloudWatch Logs dla wskazanej funkcji Lambda (przez AWS SDK v3) i wysyłający je do OpenAI API w celu diagnozy. Dodaj workflow GitHub Actions `.github/workflows/diagnose.yml` z triggerem `workflow_dispatch` (ręczne uruchomienie), przyjmującym parametr `function_name`. Skrypt powinien wypisać diagnozę do logu Actions oraz opublikować ją jako komentarz do ostatniego otwartego PR w repozytorium (przez Octokit).

---

### Zadanie 3 — Średnie
**Lambda agenta self-healing wyzwalana przez CloudWatch Alarm**

Utwórz funkcję Lambda `self-healing-agent` subskrybującą temat SNS powiązany z alarmem CloudWatch (próg: >3 błędy `ERROR` w ciągu 5 minut dla monitorowanej funkcji). Agent powinien:
- pobierać logi CloudWatch z ostatnich 30 minut,
- wysyłać je do OpenAI z prośbą o diagnozę,
- jeśli kategoria to `config` (np. timeout) — automatycznie zaktualizować konfigurację Lambda przez SDK,
- jeśli kategoria to `code` — wysłać powiadomienie email przez SNS z opisem problemu.

Skonfiguruj politykę IAM z minimalnymi uprawnieniami. Przetestuj działanie, celowo wywołując timeout w monitorowanej funkcji (np. przez `await new Promise(r => setTimeout(r, 30000))`).

---

### Zadanie 4 — Zaawansowane
**Autonomiczny agent self-healing z pętlą weryfikacji**

Zbuduj zaawansowanego agenta self-healing z pętlą weryfikacji po zastosowaniu poprawki:
1. Agent pobiera logi, diagnozuje problem i stosuje poprawkę konfiguracyjną (np. zwiększa timeout Lambda).
2. Po 5 minutach agent ponownie pobiera logi i sprawdza, czy błędy ustąpiły — jeśli tak, zapisuje do MongoDB sukces naprawy (`healing_log`); jeśli nie, eskaluje poprzez otwarcie PR na GitHubie z pełnym raportem diagnostycznym.
3. Cały stan pętli weryfikacji jest koordynowany przez **AWS Step Functions** (maszyna stanów składająca się z kroków: `Diagnose → ApplyFix → Wait5min → Verify → Success|Escalate`).
4. Zbuduj komponent Angular Material `/admin/healing` wyświetlający historię zdarzeń self-healing z kolekcji `healing_log` w MongoDB: datę, opis problemu, zastosowaną poprawkę, wynik weryfikacji i link do PR (jeśli dotyczy).
5. Dodaj do `README.md` repozytorium diagram przepływu agenta (ASCII art lub Mermaid) opisujący architekturę self-healing.
