# 4.4 Deployment i Monitoring: Wdrażanie na AWS Lambda (serverless) oraz monitorowanie kosztów i jakości odpowiedzi AI

**Moduł 4: Szybkie Wdrażanie (Ship Fast) w oparciu o Angular**

## Teoria

### 1. Architektura serverless — czym jest AWS Lambda i kiedy ją stosować?

**AWS Lambda** to usługa obliczeniowa działająca w modelu **Function as a Service (FaaS)**. Zamiast utrzymywać własny serwer (EC2, VPS), piszesz funkcję, która uruchamia się na żądanie. AWS automatycznie zarządza infrastrukturą: skalowaniem, dostępnością i aktualizacjami systemu operacyjnego.

Model rozliczenia: płacisz wyłącznie za **czas wykonania** funkcji (z dokładnością do milisekund) i **liczbę wywołań**. Przy umiarkowanym ruchu (do ~1 miliona wywołań miesięcznie) AWS Lambda mieści się w bezpłatnym tierze.

Architektura wdrożenia aplikacji AI z poprzednich sekcji:

```
[Przeglądarka / Angular SPA]
        │  HTTPS
        ▼
[AWS CloudFront]  ──────────────────────────────────────────────────────────┐
        │                                                                    │
  statyczne pliki (S3)                                               [S3 bucket]
        │                                                           (Angular build)
        │  /api/*
        ▼
[AWS API Gateway]
        │
        ▼
[AWS Lambda]  (Node.js runtime)
        │
        ├──► [OpenAI / Anthropic API]
        └──► [MongoDB Atlas]
```

Lambda jest idealna dla **backendu AI**, bo:
- Wywołania są sporadyczne (użytkownik wysyła pytanie, czeka na odpowiedź) — brak ciągłego ruchu.
- Czas zimnego startu (ang. *cold start*) Node.js wynosi ~200–500 ms — akceptowalny dla czatbota.
- Automatyczne skalowanie: jeśli 1000 użytkowników pyta jednocześnie, AWS uruchamia 1000 równoległych instancji.

### 2. Przygotowanie Node.js do wdrożenia jako Lambda

Lambda oczekuje eksportowanej funkcji `handler`, przyjmującej obiekt `event` (żądanie HTTP) i `context`. Dzięki bibliotece **`@aws-sdk`** lub adapterom (`@codegenie/serverless-express`) możemy łatwo przenieść istniejącą aplikację Express na Lambda.

**Adapter serverless-express — minimalna zmiana w kodzie:**

```javascript
// lambda.js — punkt wejścia dla AWS Lambda
import serverlessExpress from '@codegenie/serverless-express';
import app from './app.js';   // istniejąca aplikacja Express

export const handler = serverlessExpress({ app });
```

```javascript
// app.js — Express bez app.listen()
import express from 'express';
import { connectDB } from './db.js';
import chatRouter from './routes/chat.js';

const app = express();
app.use(express.json());
app.use('/api/chat', chatRouter);

await connectDB();   // połączenie z MongoDB Atlas

export default app;
```

**Instalacja:**

```bash
npm install @codegenie/serverless-express
```

Lambda reużywa kontenera między wywołaniami (ang. *warm instance*) — połączenie z MongoDB nawiązane przy pierwszym wywołaniu pozostaje otwarte dla kolejnych, co eliminuje narzut inicjalizacji.

### 3. Wdrożenie przez GitHub Actions — CI/CD pipeline

Automatyczne wdrożenie na Lambda przy każdym `push` do gałęzi `main` to wzorzec gwarantujący spójność między kodem w repozytorium a produkcją.

**Struktura pipeline'u:**

```yaml
# .github/workflows/deploy.yml
name: Deploy to AWS Lambda

on:
  push:
    branches: [main]

jobs:
  deploy-backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
          cache-dependency-path: backend/package-lock.json

      - name: Install dependencies
        working-directory: backend
        run: npm ci --omit=dev

      - name: Create deployment package
        working-directory: backend
        run: zip -r ../function.zip . -x "*.test.js" "*.spec.js"

      - name: Deploy to Lambda
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: eu-west-1

      - name: Update Lambda function
        run: |
          aws lambda update-function-code \
            --function-name ai-app-backend \
            --zip-file fileb://function.zip

  deploy-frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22'

      - name: Build Angular
        working-directory: frontend
        run: |
          npm ci
          npx ng build --configuration production

      - name: Deploy to S3
        run: |
          aws s3 sync frontend/dist/my-ai-app/browser s3://${{ secrets.S3_BUCKET }} \
            --delete --cache-control "max-age=31536000,immutable"

      - name: Invalidate CloudFront
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CF_DISTRIBUTION_ID }} \
            --paths "/*"
```

Sekrety (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `S3_BUCKET`, `CF_DISTRIBUTION_ID`) definiujemy w ustawieniach repozytorium GitHub: **Settings → Secrets and variables → Actions** — nigdy bezpośrednio w pliku YAML.

### 4. Zmienne środowiskowe i sekrety w Lambda

Lambda udostępnia zmienne środowiskowe konfigurowane przez konsolę AWS lub CLI. Wrażliwe dane (klucze API, connection string MongoDB) najlepiej przechowywać w **AWS Secrets Manager** i pobierać je w kodzie podczas inicjalizacji.

**Ustawienie przez AWS CLI:**

```bash
aws lambda update-function-configuration \
  --function-name ai-app-backend \
  --environment "Variables={
    NODE_ENV=production,
    MONGODB_URI=mongodb+srv://...,
    OPENAI_API_KEY=sk-...,
    JWT_SECRET=...
  }"
```

**Bezpieczniejsza alternatywa — AWS Secrets Manager:**

```javascript
// secrets.js — pobieranie sekretów przy zimnym starcie
import { SecretsManagerClient, GetSecretValueCommand } from '@aws-sdk/client-secrets-manager';

const client = new SecretsManagerClient({ region: 'eu-west-1' });
let cachedSecrets;

export async function getSecrets() {
  if (cachedSecrets) return cachedSecrets;  // cache na czas życia kontenera

  const response = await client.send(
    new GetSecretValueCommand({ SecretId: 'ai-app/production' })
  );
  cachedSecrets = JSON.parse(response.SecretString);
  return cachedSecrets;
}
```

Cachowanie sekretów w zmiennej modułu jest kluczowe — wywołanie Secrets Manager kosztuje (API call) i wydłuża zimny start.

### 5. Monitorowanie kosztów wywołań AI

Koszty aplikacji AI mogą rosnąć nieoczekiwanie szybko — szczególnie przy tokenie GPT-4o wycenianych per request. Kluczowe jest **aktywne monitorowanie** od pierwszego dnia.

**Metadane kosztów z odpowiedzi OpenAI:**

```javascript
const completion = await openai.chat.completions.create({
  model: 'gpt-4o',
  messages: [ /* ... */ ],
});

const { prompt_tokens, completion_tokens, total_tokens } = completion.usage;

// Zapis metryk do MongoDB
await db.collection('usage_logs').insertOne({
  sessionId,
  userId: req.userId,
  model: 'gpt-4o',
  promptTokens: prompt_tokens,
  completionTokens: completion_tokens,
  totalTokens: total_tokens,
  // Koszt wg cennika OpenAI (kwiecień 2026)
  estimatedCostUsd: (prompt_tokens * 0.0000025) + (completion_tokens * 0.00001),
  ts: new Date(),
});
```

**Agregacja dziennych kosztów (MongoDB):**

```javascript
const dailyCosts = await db.collection('usage_logs').aggregate([
  {
    $group: {
      _id: { $dateToString: { format: '%Y-%m-%d', date: '$ts' } },
      totalTokens: { $sum: '$totalTokens' },
      totalCostUsd: { $sum: '$estimatedCostUsd' },
      requests: { $count: {} },
    },
  },
  { $sort: { _id: -1 } },
  { $limit: 30 },
]).toArray();
```

Dodatkowe narzędzia monitorowania kosztów:
- **AWS Cost Explorer** — koszty Lambda, API Gateway, S3, CloudFront.
- **AWS Budgets** — alerty email/SMS po przekroczeniu progu (np. $50/miesiąc).
- **OpenAI Usage Dashboard** — oficjalny panel zużycia tokenów i kosztów.

### 6. Monitorowanie jakości i logowanie w Lambda

Lambda integruje się natywnie z **AWS CloudWatch** — wszystkie logi z `console.log()` trafiają automatycznie do CloudWatch Logs bez żadnej konfiguracji.

**Strukturalne logowanie (JSON):**

```javascript
// logger.js
export function log(event, data = {}) {
  console.log(JSON.stringify({
    event,
    ...data,
    ts: new Date().toISOString(),
    requestId: process.env.AWS_REQUEST_ID,
  }));
}
```

```javascript
// Użycie w routerze
log('chat_request', { userId: req.userId, model: 'gpt-4o', tokens: total_tokens });
log('chat_error', { userId: req.userId, error: err.message });
```

Strukturalne logi (JSON) pozwalają na łatwe filtrowanie i tworzenie metryk w CloudWatch:

```bash
# Zapytanie CloudWatch Logs Insights — błędy z ostatnich 24h
fields @timestamp, event, error
| filter event = "chat_error"
| sort @timestamp desc
| limit 50
```

**Alarmy CloudWatch** informują o problemach zanim użytkownik je zgłosi:

```bash
# Alarm: więcej niż 5 błędów na minutę
aws cloudwatch put-metric-alarm \
  --alarm-name "LambdaErrorsHigh" \
  --metric-name Errors \
  --namespace AWS/Lambda \
  --dimensions Name=FunctionName,Value=ai-app-backend \
  --statistic Sum \
  --period 60 \
  --threshold 5 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 1 \
  --alarm-actions arn:aws:sns:eu-west-1:123456789:AlertsTopic
```

## Zadania praktyczne

### Zadanie 1 — Łatwe
**Pierwszy build produkcyjny Angular i wdrożenie na S3**

Wykonaj `ng build --configuration production` dla aplikacji Angular z poprzednich sekcji. Utwórz bucket S3 z włączoną opcją „Static website hosting" i wgraj do niego zawartość folderu `dist/`. Skonfiguruj politykę publicznego dostępu do bucketu (`GetObject` dla wszystkich). Zweryfikuj, że aplikacja Angular działa poprawnie pod adresem URL endpointu S3.

---

### Zadanie 2 — Średnie
**Wdrożenie backendu Node.js jako AWS Lambda**

Przystosuj backend Express z sekcji 4.2 do uruchamiania jako Lambda przy użyciu adaptera `@codegenie/serverless-express`. Utwórz funkcję Lambda w konsoli AWS (runtime Node.js 22.x) i wgraj kod ręcznie jako plik ZIP. Skonfiguruj AWS API Gateway (HTTP API) jako trigger funkcji i przetestuj działanie endpointu `POST /api/chat` przez `curl`. Dodaj zmienne środowiskowe (`OPENAI_API_KEY`, `MONGODB_URI`) w konfiguracji funkcji Lambda.

---

### Zadanie 3 — Średnie
**Pipeline CI/CD GitHub Actions — deploy Lambda + S3**

Utwórz plik `.github/workflows/deploy.yml` automatyzujący wdrożenie: budowanie Angular i synchronizację z S3 (job `deploy-frontend`) oraz pakowanie i aktualizację kodu Lambda (job `deploy-backend`). Skonfiguruj sekrety GitHub Actions (klucze AWS, nazwa bucketu, ID dystrybucji CloudFront). Przetestuj pipeline wykonując `push` na gałąź `main` i obserwuj logi w zakładce GitHub Actions.

### Zadanie 4 — Zaawansowane
**Dashboard monitorowania kosztów AI i alertowanie**

Zaimplementuj pełny system monitorowania kosztów end-to-end:
1. Rozbuduj backend Lambda o logowanie metadanych każdego wywołania OpenAI (tokeny, szacowany koszt, `userId`, model) do kolekcji `usage_logs` w MongoDB.
2. Utwórz endpoint `GET /api/admin/usage?from=2026-04-01&to=2026-04-30` zwracający agregację dziennych kosztów i liczby żądań.
3. Zbuduj komponent Angular Material z wykresem kosztów dziennych (użyj `<mat-table>` lub biblioteki `Chart.js`) widoczny pod `/admin/usage` — chroniony guardem Angular sprawdzającym rolę `admin` w JWT.
4. Skonfiguruj alarm AWS CloudWatch wysyłający email (SNS) gdy liczba błędów Lambda przekroczy 10 w ciągu minuty.
5. Do repozytorium GitHub dodaj plik `RUNBOOK.md` opisujący: jak interpretować alarmy, jak sprawdzać logi w CloudWatch Logs Insights i jak reagować na przekroczenie budżetu AI.
