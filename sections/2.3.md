# 2.3 Narzędzia Agentowe CLI: Praca z Aider i Open Interpreter – programowanie przez dialog w konsoli

> **Moduł 2: Ekosystem Vibe Coding i AI-first IDE**

---

## Teoria

### 1. Narzędzia agentowe CLI – czym są i dlaczego warto je znać?

Narzędzia agentowe CLI (ang. *Command-Line Interface agent tools*) to kategoria programów uruchamianych w terminalu, które integrują modele językowe bezpośrednio z powłoką systemową. W odróżnieniu od IDE-based assistants (Cursor, Copilot), narzędzia CLI działają niezależnie od edytora – można je uruchomić w każdym środowisku z dostępem do terminala: na zdalnym serwerze SSH, w kontenerze Docker, w skryptach CI/CD, czy po prostu gdy edytor jest zbędnym narzutem.

**Kluczowe cechy narzędzi agentowych CLI:**

- **Bez GUI** – interfejs tekstowy, idealne do automatyzacji i środowisk serwerowych.
- **Bezpośredni dostęp do systemu plików** – operują na prawdziwych plikach, nie tylko edytują tekst w buforze edytora.
- **Integracja z Git** – narzędzia takie jak Aider automatycznie commitują zmiany, tworząc audytowalną historię pracy AI.
- **Model-agnostyczne** – obsługują modele OpenAI, Anthropic, lokalne modele (Ollama, Llama), co daje pełną kontrolę nad kosztami i prywatnością.

Analogia: jeśli Cursor to pełne środowisko pracy (jak warsztat stolarski z wszystkimi narzędziami), to narzędzia CLI to precyzyjne, przenośne narzędzia (jak elektryczna wiertarka) – mniejsze, szybsze w użyciu i niezależne od kontekstu.

### 2. Aider – programowanie przez dialog z automatycznym commitowaniem

**Aider** (z ang. *AI pair programmer in your terminal*) to narzędzie open-source pozwalające na programowanie w stylu dialogowym bezpośrednio w terminalu. Jego wyróżniającą cechą jest **głęboka integracja z Git** – każda zmiana kodu zaakceptowana przez użytkownika jest natychmiast commitowana z automatycznie wygenerowaną wiadomością commita.

**Instalacja i uruchomienie:**

```bash
# Instalacja przez pip
pip install aider-chat

# Uruchomienie z modelem Claude (zalecane dla kodu)
export ANTHROPIC_API_KEY=sk-ant-...
aider --model claude-3-5-sonnet-20241022

# Lub z GPT-4o
export OPENAI_API_KEY=sk-...
aider --model gpt-4o

# Uruchomienie z automatycznym wskazaniem plików
aider src/services/user.service.js src/models/user.model.js
```

**Podstawowe komendy w sesji Aidra:**

| Komenda | Opis |
|---|---|
| `/add plik.js` | Dodaje plik do aktywnego kontekstu sesji |
| `/drop plik.js` | Usuwa plik z kontekstu |
| `/diff` | Pokazuje bieżące niezacommitowane zmiany |
| `/undo` | Cofa ostatni commit stworzony przez Aidra |
| `/run polecenie` | Uruchamia polecenie powłoki i pokazuje wynik modelowi |
| `/ask pytanie` | Zadaje pytanie bez modyfikowania kodu |
| `/exit` | Kończy sesję |

**Przykładowa sesja:**

```bash
$ aider src/routes/auth.js src/services/auth.service.js

Aider v0.50.0
> dodaj endpoint POST /auth/logout który usuwa sesję z MongoDB i zwraca 200

# Aider pokazuje diff zmian, pyta o akceptację, commituje:
# "feat: add POST /auth/logout endpoint with session cleanup"
```

Automatyczne commitowanie jest szczególnie cenne w środowiskach zespołowych – historia Git pokazuje dokładnie co AI zmieniło i kiedy, co ułatwia przeglądy kodu i rollback.

### 3. Aider – zaawansowane tryby pracy i konfiguracja

Aider oferuje kilka trybów pracy dostosowanych do różnych scenariuszy:

**Tryby edycji (`--edit-format`):**

- **`diff`** (domyślny) – model generuje ujednolicone diffy (unified diff), precyzyjny i oszczędny na tokenach.
- **`whole`** – model przepisuje cały plik; lepszy gdy zmiany są rozległe.
- **`udiff`** – rozszerzony format diff, przydatny przy skomplikowanych zmianach.

**Tryb Architect + Editor** (`--architect`): Aider uruchamia dwa modele równolegle – mocniejszy model (np. `o1`) planuje architekturę zmian, słabszy (np. `claude-3-5-haiku`) implementuje szczegóły. To rozwiązanie optymalizuje stosunek jakości do kosztu.

```bash
aider --architect --model o1 --editor-model claude-3-5-haiku-20241022
```

**Plik konfiguracyjny `.aider.conf.yml`:**

```yaml
# .aider.conf.yml – zapisany w katalogu projektu
model: claude-3-5-sonnet-20241022
auto-commits: true
dirty-commits: false
show-diffs: true
read: # pliki zawsze dołączane jako kontekst tylko do odczytu
  - README.md
  - docs/architecture.md
```

**Tryb `--no-auto-commits`** przydatny gdy wolisz samodzielnie kontrolować co trafia do historii Git – Aider nadal modyfikuje pliki, ale nie commituje bez Twojej zgody.

### 4. Open Interpreter – uruchamianie kodu przez model AI

**Open Interpreter** (projekt open-source, znany też jako `interpreter`) to narzędzie CLI umożliwiające modelowi językowemu **pisanie i wykonywanie kodu** w środowisku użytkownika. W odróżnieniu od Aidra, który koncentruje się na edycji plików projektu, Open Interpreter może uruchamiać dowolne skrypty, przetwarzać dane, instalować pakiety i operować na systemie plików – wszystko w odpowiedzi na polecenia w języku naturalnym.

**Instalacja i uruchomienie:**

```bash
# Instalacja
pip install open-interpreter

# Uruchomienie (interaktywna sesja)
interpreter

# Uruchomienie z konkretnym modelem
interpreter --model gpt-4o

# Uruchomienie w trybie lokalnym (bez wysyłania kodu na serwery)
interpreter --local --model ollama/llama3
```

**Przykładowe zastosowania:**

```
> Pobierz dane z https://api.github.com/repos/angular/angular/releases,
  wyodrębnij 5 ostatnich wersji z datami i zapisz do pliku releases.json

# Open Interpreter napisze i wykona kod JavaScript/Bash/Python realizujący to zadanie
```

```
> Znajdź wszystkie pliki .log w folderze /var/log starsze niż 30 dni
  i skompresuj je do archiwum tar.gz z datą w nazwie

# Model generuje skrypt Bash, pyta o potwierdzenie przed wykonaniem
```

**Ważna różnica w stosunku do Aidra:** Open Interpreter *wykonuje* kod, nie tylko go edytuje. Oznacza to, że może bezpowrotnie modyfikować pliki, usuwać dane lub wykonywać żądania sieciowe. Zawsze weryfikuj kod przed akceptacją wykonania.

### 5. Bezpieczeństwo i ograniczenia narzędzi CLI

Praca z narzędziami agentowymi CLI wiąże się z unikalnym zestawem zagrożeń bezpieczeństwa, które różnią się od ryzyk typowych dla edytorów kodu.

**Główne ryzyka:**

| Ryzyko | Narzędzie | Mitygacja |
|---|---|---|
| Przypadkowe usunięcie plików | Open Interpreter | Używaj w izolowanym środowisku (Docker, VM) |
| Wyciek kluczy API przez kontekst | Aider, OI | Dodaj `.env` do `.aiderignore`, nigdy nie dodawaj do kontekstu |
| Nadpisanie plików produkcyjnych | Aider | Pracuj na osobnej gałęzi Git (`git checkout -b ai/feature`) |
| Wykonanie złośliwego kodu w odpowiedzi modelu | Open Interpreter | Czytaj każdy snippet przed akceptacją |
| Prompt injection przez dane zewnętrzne | OI | Nie przekazuj niezaufanych danych jako kontekstu |

**Dobre praktyki:**

```bash
# Zawsze pracuj na gałęzi feature, nie na main
git checkout -b ai/refactor-auth-module

# Aider – jawnie wyklucz wrażliwe pliki
echo ".env\n.env.local\nsecrets.json" >> .aiderignore

# Open Interpreter – tryb bezpieczny (pyta przed każdym wykonaniem)
interpreter --safe-mode ask

# Używaj lokalnych modeli dla wrażliwego kodu
interpreter --local --model ollama/codellama
```

**Środowisko izolowane z Dockerem** – dla zadań szczególnie ryzykownych (np. automatyzacja systemu plików, operacje na bazie danych):

```bash
docker run -it --rm \
  -v $(pwd):/workspace \
  -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY \
  python:3.12-slim \
  bash -c "pip install aider-chat && cd /workspace && aider"
```

### 6. Kiedy używać CLI vs. IDE-based tools – praktyczne porównanie

Wybór między narzędziami CLI a IDE-based asystentami nie jest zero-jedynkowy – doświadczony developer korzysta z obu, dopasowując narzędzie do kontekstu zadania.

| Kryterium | Aider / Open Interpreter | Cursor / Copilot |
|---|---|---|
| Środowisko | Terminal, SSH, CI/CD, Docker | VS Code, JetBrains |
| Integracja z Git | Automatyczna (Aider) | Manualna |
| Wieloplikowe edycje | ✓ | ✓ |
| Wykonywanie kodu | ✓ (OI) | Ograniczone |
| Wizualizacja diff | Tekstowy diff | Graficzny diff |
| Koszt | Zależny od API | Subskrypcja + API |
| Prywatność | Modele lokalne możliwe | Wymaga połączenia |

**Rekomendowane przypadki użycia CLI:**

- **Migracje danych** – skrypt jednorazowy, nie wymaga IDE.
- **Automatyzacja w CI/CD** – Aider może być uruchamiany jako krok pipeline'u do automatycznego generowania lub poprawiania kodu.
- **Środowiska serwerowe** – praca na zdalnych maszynach bez GUI.
- **Szybkie skrypty** – Open Interpreter do jednorazowych zadań przetwarzania danych.

```bash
# Przykład: Aider w GitHub Actions do automatycznej naprawy lintingu
- name: Fix lint errors with AI
  run: |
    pip install aider-chat
    aider --model gpt-4o-mini --no-auto-commits \
      --message "Napraw wszystkie błędy ESLint w plikach zmienionych w tym PR" \
      $(git diff --name-only HEAD~1 | grep '\.js$')
```

## Zadania praktyczne

### Zadanie 1 — Łatwe: Pierwsza sesja z Aiderem

**Temat:** Zainstaluj Aider w lokalnym środowisku (`pip install aider-chat`), skonfiguruj klucz API (OpenAI lub Anthropic) i przeprowadź pierwszą sesję programistyczną. Wskaż dwa istniejące pliki projektu Node.js (np. plik routera i serwisu), a następnie poproś Aidra o dodanie walidacji danych wejściowych do jednej funkcji. Zaobserwuj jak Aider prezentuje diff, commituje zmianę i generuje wiadomość commita. Użyj `/undo` aby cofnąć commit i powtórz operację z innym sformułowaniem prompta – porównaj jakość obu wyników.

---

### Zadanie 2 — Średnie: Refaktoryzacja modułu z automatycznym zatwierdzaniem

**Temat:** Skonfiguruj plik `.aider.conf.yml` dla projektu Node.js + MongoDB z preferowanym modelem, włączonymi auto-commitami i plikiem `README.md` jako kontekstem tylko do odczytu. Następnie przeprowadź wieloetapową refaktoryzację wybranego modułu (min. 3 pliki): zmień obsługę błędów z `try/catch` wewnątrz handlerów na centralny middleware Express, dodaj walidację wejść przez bibliotekę `joi` lub `zod`. Po zakończeniu przejrzyj historię Git (`git log --oneline`) i oceń jakość automatycznie wygenerowanych wiadomości commitów.

---

### Zadanie 3 — Średnie: Automatyzacja zadania z Open Interpreter

**Temat:** Uruchom Open Interpreter i zleć mu w języku naturalnym wykonanie następującego zadania analitycznego: pobierz listę publicznych repozytoriów wybranej organizacji z GitHub API (bez autoryzacji, limit 30 repozytoriów), wyodrębnij nazwę, liczbę gwiazdek i datę ostatniego push, posortuj według gwiazdek malejąco i zapisz wynik do pliku `github-repos.json` oraz wygeneruj prosty raport tekstowy `github-repos-report.txt`. Obserwuj każdy krok generowania i wykonywania kodu przez model. Przed akceptacją każdego snippetu weryfikuj czy nie wykonuje nieoczekiwanych operacji.

---

### Zadanie 4 — Zaawansowane: Integracja Aidra w pipeline CI/CD na GitHubie

**Temat:** Zaprojektuj i zaimplementuj workflow GitHub Actions (`.github/workflows/ai-fix.yml`), który uruchamia się po nieudanym biegu testów na gałęzi `development`. Workflow powinien: zainstalować Aider, pobrać logi błędów z nieudanego testu, uruchomić Aidra w trybie nieinteraktywnym (`--message`) z poleceniem naprawy błędu na podstawie logu, zacommitować poprawki i opcjonalnie otworzyć PR z naprawą. Zabezpiecz klucz API jako GitHub Secret. Przetestuj pipeline commitując celowo uszkodzony kod. Opisz ograniczenia tego podejścia i scenariusze, w których automatyczna naprawa może zawieść lub wprowadzić regresję.
