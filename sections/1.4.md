# 1.4 API First: Integracja z OpenAI API i Anthropic API (autoryzacja, koszty, parametry temperature/top_p)

> **Moduł 1: Fundamenty i Mechanizmy LLM**

---

## Teoria

### 1. Podejście API First — dlaczego integracja przez API to fundament AI developmentu

**API First** to paradygmat tworzenia oprogramowania, w którym interfejs programistyczny (API) traktuje się jako produkt pierwszej klasy — projektuje się go i dokumentuje przed implementacją pozostałych warstw systemu. W kontekście modeli językowych oznacza to świadome odseparowanie inteligencji (modelu) od aplikacji (interfejsu użytkownika, logiki biznesowej, bazy danych) i komunikowanie się z nią wyłącznie przez dobrze zdefiniowany kontrakt HTTP.

Dlaczego to podejście ma takie znaczenie? Ponieważ model językowy jest **zewnętrzną, wymienialną usługą**. Dziś korzystasz z GPT-4o, jutro może się okazać, że Claude 3.5 Sonnet jest tańszy i dokładniejszy dla Twojego przypadku użycia — zmiana powinna wymagać korekty kilku linii kodu, nie przepisania całej aplikacji. Dobrze zaprojektowana warstwa abstrakcji nad API LLM to różnica między projektem, który można utrzymywać, a projektem, który po roku staje się nie do ruszenia.

**Architektura typowej integracji API First z LLM:**

```
Aplikacja (Angular)
       ↓ HTTP
Backend / API Gateway (Node.js + Express)  ←→  Klucze API (env vars / AWS Secrets Manager)
       ↓ HTTPS
  ┌────────────┐    ┌───────────────────┐
  │ OpenAI API │    │ Anthropic API     │
  │ api.openai │    │ api.anthropic.com │
  └────────────┘    └───────────────────┘
```

Kluczowa zasada bezpieczeństwa: **klucze API nigdy nie opuszczają backendu**. Jeśli piszesz aplikację Angular i wywołujesz API LLM bezpośrednio z przeglądarki, narażasz klucz na kradzież przez każdego, kto otworzy DevTools. Właściwym rozwiązaniem jest backendowy proxy — własny endpoint `/api/chat`, który trzyma klucz po stronie serwera.

**Podstawowe pojęcia:**
- **Endpoint** — konkretny adres URL przyjmujący żądania, np. `https://api.openai.com/v1/chat/completions`
- **Request body** — dane wysyłane do API (model, wiadomości, parametry)
- **Response** — odpowiedź API zawierająca wygenerowany tekst, metadane i informacje o zużyciu tokenów
- **Rate limit** — limit liczby żądań lub tokenów na jednostkę czasu, nakładany przez dostawcę

### 2. Autoryzacja — klucze API, zmienne środowiskowe i dobre praktyki bezpieczeństwa

Zarówno OpenAI, jak i Anthropic używają **Bearer token authentication** — wysyłasz klucz w nagłówku HTTP `Authorization`. Jest to standard OAuth 2.0 dla klientów nieinteraktywnych i oznacza, że każde żądanie musi zawierać poprawny klucz, by zostać zautoryzowane.

**Jak pobrać klucze:**
- **OpenAI:** `platform.openai.com` → Settings → API Keys → „Create new secret key"
- **Anthropic:** `console.anthropic.com` → Settings → API Keys → „Create Key"

Oba klucze pokazywane są tylko raz — po zamknięciu okna nie możesz ich odtworzyć, jedynie unieważnić i wygenerować nowy.

**Schemat nagłówków:**

| Dostawca | Nagłówek autoryzacji | Dodatkowe nagłówki |
|---|---|---|
| OpenAI | `Authorization: Bearer sk-...` | (opcjonalnie) `OpenAI-Organization: org-...` |
| Anthropic | `x-api-key: sk-ant-...` | `anthropic-version: 2023-06-01` (wymagany) |

**Przechowywanie kluczy w projekcie Node.js:**

```bash
# .env (nigdy nie commituj tego pliku!)
OPENAI_API_KEY=sk-proj-...
ANTHROPIC_API_KEY=sk-ant-api03-...
```

```javascript
// server.js (Node.js + Express)
import 'dotenv/config';
import OpenAI from 'openai';
import Anthropic from '@anthropic-ai/sdk';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });
```

```bash
# .gitignore — absolutne minimum
.env
.env.local
.env.*.local
node_modules/
```

**W środowiskach produkcyjnych** (AWS) klucze przechowuj w dedykowanych usługach: AWS Secrets Manager lub AWS Parameter Store — nigdy w `docker-compose.yml` ani w kodzie.

```bash
# Dodawanie klucza do AWS Parameter Store (SSM)
aws ssm put-parameter \
  --name "/myapp/prod/OPENAI_API_KEY" \
  --value "sk-proj-..." \
  --type "SecureString" \
  --region eu-central-1
```

Praktyczna zasada: **traktuj klucz API jak hasło do konta bankowego**. Rotuj klucze co kilka miesięcy, twórz osobne klucze dla każdego środowiska (dev/staging/prod) i ogranicz ich uprawnienia do niezbędnego minimum (jeśli dostawca to umożliwia).

### 3. Anatomia żądania — struktura wywołania Chat Completions API

Oba API — OpenAI i Anthropic — opierają się na modelu **wieloturowej konwersacji**, w której każde żądanie zawiera pełną historię wymiany wiadomości. To ważna różnica w stosunku do prostych API tekstowych: model nie ma wbudowanej pamięci — to Ty, jako deweloper, przechowujesz historię i przesyłasz ją przy każdym wywołaniu.

**OpenAI Chat Completions — minimalne żądanie (Node.js):**

```javascript
import OpenAI from 'openai';
const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

const response = await openai.chat.completions.create({
  model: 'gpt-4o',
  messages: [
    { role: 'system', content: 'Jesteś pomocnym asystentem programisty.' },
    { role: 'user',   content: 'Wyjaśnij różnicę między Promise a async/await.' }
  ],
  max_tokens: 512,
  temperature: 0.7,
});

console.log(response.choices[0].message.content);
// Zużycie tokenów:
console.log(response.usage); // { prompt_tokens, completion_tokens, total_tokens }
```

**Anthropic Messages API — odpowiednik (Node.js):**

```javascript
import Anthropic from '@anthropic-ai/sdk';
const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

const response = await anthropic.messages.create({
  model: 'claude-3-5-sonnet-20241022',
  max_tokens: 512,
  system: 'Jesteś pomocnym asystentem programisty.',   // system prompt — osobne pole
  messages: [
    { role: 'user', content: 'Wyjaśnij różnicę między Promise a async/await.' }
  ],
});

console.log(response.content[0].text);
console.log(response.usage); // { input_tokens, output_tokens }
```

**Kluczowe różnice strukturalne między API:**

| Aspekt | OpenAI | Anthropic |
|---|---|---|
| System prompt | Jako wiadomość `{ role: 'system', ... }` | Osobne pole `system: "..."` (top-level) |
| Odpowiedź | `response.choices[0].message.content` | `response.content[0].text` |
| Tokeny wejściowe | `response.usage.prompt_tokens` | `response.usage.input_tokens` |
| Streaming | `stream: true` | `stream: true` (te same zasady) |
| Wersjonowanie | Przez nazwę modelu | Nagłówek `anthropic-version` (wymagany) |

Warto też zapamiętać: **wiadomości w historii muszą być naprzemienne** (user → assistant → user → …). Wysłanie dwóch wiadomości `user` pod rząd spowoduje błąd walidacji w API Anthropic.

### 4. Parametry generowania — temperature, top_p i ich wpływ na jakość odpowiedzi

Parametry generowania to pokrętła, którymi deweloper kontroluje zachowanie modelu podczas próbkowania kolejnych tokenów. Zrozumienie ich działania jest niezbędne do świadomego kompromisu między **kreatywnością a przewidywalnością** odpowiedzi.

**Temperature (temperatura)**

Temperature skaluje rozkład prawdopodobieństwa nad słownikiem tokenów przed próbkowaniem. Intuicja:

- **temperature = 0.0** — model zawsze wybiera token z najwyższym prawdopodobieństwem (zachowanie deterministyczne). Idealne do zadań faktograficznych, ekstrakcji danych, parsowania, generowania kodu.
- **temperature ≈ 0.7** — złoty środek dla większości zastosowań asystenta. Model jest konkretny, ale nie sztywny.
- **temperature ≈ 1.2–2.0** — wysoka losowość, odpowiedź bywa nie do przewidzenia. Używane w kreatywnym pisaniu, brainstormingu.

Ważna zasada: **nie używaj wysokiej temperatury i top_p jednocześnie** — modele są zwykle kalibrowane do pracy z jednym z tych parametrów ustawionym blisko wartości domyślnej.

**Top-p (nucleus sampling)**

Top-p definiuje minimalny kumulatywny zakres prawdopodobieństwa, z którego model próbkuje następny token. Przy `top_p = 0.9` model bierze pod uwagę tylko te tokeny, które łącznie odpowiadają za 90% masy prawdopodobieństwa — odcinając ogon mało prawdopodobnych (i często dziwnych) tokenów.

**Tabela praktycznych ustawień:**

| Zastosowanie | temperature | top_p | Uzasadnienie |
|---|---|---|---|
| Generowanie kodu | 0.0 – 0.2 | 1.0 | Deterministyczność, poprawność składniowa |
| Ekstrakcja danych / JSON | 0.0 | 1.0 | Zero halucynacji, ścisły format |
| Q&A / asystent | 0.7 | 1.0 | Naturalny, konkretny styl |
| Pisanie copywritingu | 1.0 | 0.95 | Różnorodność sformułowań |
| Kreatywne pisanie | 1.2 – 1.5 | 0.9 | Nieoczekiwane połączenia idei |

**Inne ważne parametry:**
- **`max_tokens`** — twardy limit tokenów odpowiedzi. Pamiętaj, że niezużyte tokeny *nie są* naliczane — limit nie jest gwarancją długości
- **`stop`** — lista ciągów znaków, przy których generowanie jest przerywane (np. `["###", "\n\n"]`), przydatne przy generowaniu strukturyzowanego tekstu
- **`frequency_penalty`** / **`presence_penalty`** (OpenAI) — kary za powtarzanie tokenów; zmniejszają tendencję modelu do zapętlania podobnych fraz

### 5. Koszty — model rozliczania, kalkulacja i strategie optymalizacji

LLM API są rozliczane w modelu **pay-per-token** — płacisz za każdy przetworzony token wejściowy (prompt) i każdy wygenerowany token wyjściowy (completion). Zrozumienie struktury kosztów jest kluczowe dla projektowania ekonomicznie opłacalnych aplikacji.

**Szacunkowe ceny (kwiecień 2025, sprawdź bieżące ceny w dokumentacji dostawcy):**

| Model | Tokeny wejściowe (/ 1M) | Tokeny wyjściowe (/ 1M) |
|---|---|---|
| GPT-4o | ~$2.50 | ~$10.00 |
| GPT-4o mini | ~$0.15 | ~$0.60 |
| Claude 3.5 Sonnet | ~$3.00 | ~$15.00 |
| Claude 3 Haiku | ~$0.25 | ~$1.25 |

Obserwacja: tokeny wyjściowe kosztują zazwyczaj **3–5× więcej** niż wejściowe. Każde zbędne słowo w odpowiedzi to realny koszt.

**Jak szacować koszt przed wdrożeniem:**

```javascript
// Szacowanie kosztu żądania (model GPT-4o)
function estimateCost(inputText, estimatedOutputTokens = 512) {
  // ~4 znaki = 1 token (angielski); ~3 znaki = 1 token (deutsch/polski)
  const inputTokens = Math.ceil(inputText.length / 3.5);
  const inputCost  = (inputTokens / 1_000_000) * 2.50;   // $2.50 / 1M tokenów
  const outputCost = (estimatedOutputTokens / 1_000_000) * 10.00;
  return { inputTokens, totalUSD: inputCost + outputCost };
}
```

**Strategie redukcji kosztów:**

1. **Dobór modelu do zadania** — nie używaj GPT-4o do klasyfikacji prostego tekstu, gdy GPT-4o mini radzi sobie równie dobrze za 20× niższą cenę
2. **Skracanie system promptów** — każdy prompt jest powtarzany przy każdym wywołaniu; 500 zbędnych tokenów systemu × 10 000 wywołań dziennie = 5M tokenów
3. **Caching (prompt caching)** — Anthropic i OpenAI oferują mechanizm cachowania powtarzalnych fragmentów kontekstu (np. długiego system promptu lub dokumentacji); cached tokeny kosztują ~10% normalnej ceny
4. **Ograniczanie `max_tokens`** — ustaw rozsądny limit; modele mają tendencję do generowania aż do limitu, jeśli nie są inaczej instruowane
5. **Streaming i przerwanie** — w przypadku UI możesz przerywać generowanie wcześniej, gdy użytkownik otrzymał wystarczające informacje

**Monitorowanie kosztów:** Ustaw limity budżetowe w dashboardach OpenAI/Anthropic oraz skorzystaj z narzędzi takich jak LangSmith lub własnych logów (zapisuj `usage.total_tokens` przy każdym wywołaniu do MongoDB lub dowolnej bazy analitycznej).

### 6. Obsługa błędów, rate limity i wzorzec retry z exponential backoff

Produkcyjna integracja z LLM API to nie tylko „happy path". Sieć jest zawodna, dostawcy mają limity, a modele czasem zwracają nieoczekiwane odpowiedzi. Solidna obsługa błędów jest cechą dojrzałego kodu.

**Typowe kody błędów HTTP:**

| Kod | Znaczenie | Działanie |
|---|---|---|
| `400 Bad Request` | Błędny format żądania (np. naprzemienność ról) | Sprawdź strukturę `messages`; nie ponów |
| `401 Unauthorized` | Nieprawidłowy klucz API | Sprawdź zmienne środowiskowe; nie ponów |
| `429 Too Many Requests` | Przekroczono rate limit | Poczekaj i ponów (backoff) |
| `500 / 503` | Błąd po stronie dostawcy | Ponów z backoffem |
| `408 / timeout` | Timeout sieci | Ponów z backoffem |

**Implementacja retry z exponential backoff (Node.js):**

```javascript
async function callWithRetry(fn, maxRetries = 3) {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      const isRetryable = err.status === 429 || err.status >= 500;
      if (!isRetryable || attempt === maxRetries) throw err;

      const delayMs = Math.min(1000 * 2 ** attempt + Math.random() * 500, 30_000);
      console.warn(`Attempt ${attempt + 1} failed (${err.status}). Retrying in ${delayMs}ms…`);
      await new Promise(res => setTimeout(res, delayMs));
    }
  }
}

// Użycie:
const result = await callWithRetry(() =>
  openai.chat.completions.create({ model: 'gpt-4o', messages, max_tokens: 512 })
);
```

**Rate limity** mają dwie wymiary: **RPM** (requests per minute) i **TPM** (tokens per minute). Nowe konta mają niskie limity (np. 500 RPM dla GPT-4o), które rosną automatycznie wraz z historią płatności lub na wniosek do supportu.

**Streaming** to kolejna kluczowa technika — zamiast czekać na pełną odpowiedź (co przy dłuższych tekstach może trwać 10–30 sekund), możesz odbierać tokeny na bieżąco i wyświetlać je użytkownikowi w czasie rzeczywistym:

```javascript
const stream = await openai.chat.completions.create({
  model: 'gpt-4o',
  messages,
  stream: true,
});

for await (const chunk of stream) {
  const delta = chunk.choices[0]?.delta?.content ?? '';
  process.stdout.write(delta); // lub push do WebSocket / SSE
}
```

Streaming nie tylko poprawia UX (użytkownik widzi odpowiedź natychmiast), ale też pozwala wcześnie przerwać generowanie, jeśli model „łapie" zły kierunek — co może obniżyć koszty tokena wyjściowego.

### 7. Różnice między OpenAI API i Anthropic API — wybór modelu do zadania

Obie platformy oferują modele najwyższej klasy, ale mają różne mocne strony, filozofię projektowania i specyficzne funkcje. Świadomy wybór między nimi (lub korzystanie z obu w zależności od zadania) jest wyróżnikiem doświadczonego AI developera.

**Porównanie kluczowych cech:**

| Cecha | OpenAI (GPT-4o) | Anthropic (Claude 3.5 Sonnet) |
|---|---|---|
| Okno kontekstowe | 128 000 tokenów | 200 000 tokenów |
| Multimodalność | Tekst, obrazy, audio, wideo | Tekst, obrazy (PDF, grafika) |
| Instruction following | Bardzo dobre | Wyjątkowe (szczególnie długie instrukcje) |
| Pisanie długich dokumentów | Dobre | Wyróżniające się |
| Function calling / tools | ✅ (dojrzały ekosystem) | ✅ (tool use) |
| Prompt caching | ✅ | ✅ (agresywniejszy mechanizm) |
| Analiza kodu | Bardzo dobry | Bardzo dobry |
| Bezpieczeństwo/content policy | Rygorystyczne filtry | Constitutional AI — kontekstowe |

**Kiedy używać którego:**
- **GPT-4o** — integracje wymagające dojrzałego ekosystemu narzędzi, aplikacje multimodalne (analiza audio/wideo), gdy priorytetem jest kompatybilność z istniejącymi bibliotekami (LangChain, LlamaIndex mają lepszy support OpenAI).
- **Claude 3.5 Sonnet** — zadania wymagające analizy bardzo długich dokumentów (200K kontekstu), gdy precyzja instruction following jest krytyczna, pisanie i redakcja długich treści.
- **GPT-4o mini / Claude 3 Haiku** — wszelkie zadania masowe: klasyfikacja, ekstrakcja prostych danych, augmentacja kodu w pipeline CI/CD — gdzie liczy się koszt na skali.

**Ujednolicona warstwa abstrakcji (Node.js):**

```javascript
// lib/llm-client.js — wzorzec adaptera
export async function chat({ provider = 'openai', model, messages, system, maxTokens = 512 }) {
  if (provider === 'openai') {
    const msgs = system ? [{ role: 'system', content: system }, ...messages] : messages;
    const res = await openai.chat.completions.create({ model, messages: msgs, max_tokens: maxTokens });
    return { text: res.choices[0].message.content, usage: res.usage };
  }
  if (provider === 'anthropic') {
    const res = await anthropic.messages.create({ model, system, messages, max_tokens: maxTokens });
    return { text: res.content[0].text, usage: res.usage };
  }
  throw new Error(`Unknown provider: ${provider}`);
}
```

Taki adapter pozwala przełączać dostawcę jedną zmianą konfiguracji i jest punktem wyjścia do bardziej zaawansowanych rozwiązań takich jak LiteLLM czy Portkey, które robią to samo na poziomie usługi.

### Zadanie 4 — Zaawansowane: Warstwa abstrakcji z retry, streamingiem i monitoringiem kosztów

Moduł powinien być testowalny jednostkowo — pokryj funkcję retry i adapter model co najmniej trzema testami.
