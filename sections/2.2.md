## Teoria

### 1. GitHub Copilot – architektura i tryby pracy

GitHub Copilot to asystent AI zintegrowany bezpośrednio z edytorem kodu (VS Code, JetBrains, Neovim), oparty na modelach OpenAI Codex/GPT-4o. W odróżnieniu od Cursora, Copilot funkcjonuje jako **rozszerzenie do istniejącego edytora**, co czyni go łatwiejszym w adopcji dla zespołów już korzystających z VS Code bez zmiany środowiska pracy.

**Główne tryby pracy:**

| Tryb | Skrót | Zastosowanie |
|---|---|---|
| **Inline suggestions** | automatyczne | Uzupełnianie kodu w trakcie pisania (ghost text) |
| **Copilot Chat** | `Ctrl+Alt+I` | Rozmowa z AI w panelu bocznym lub inline |
| **Inline Chat** | `Ctrl+I` | Szybkie zapytanie/edycja w miejscu kursora |
| **Quick Chat** | `Ctrl+Shift+Alt+L` | Jednorazowe pytanie bez otwierania panelu |
| **@workspace** | w czacie | Pytanie z kontekstem całego projektu |

**Jak działa dopasowanie sugestii inline:** Copilot wysyła do modelu aktualny plik, kilka ostatnio edytowanych plików oraz komentarze/nagłówki jako kontekst. Im bardziej opisowy jest komentarz lub nazwa funkcji poprzedzająca miejsce kursora, tym trafniejsza jest propozycja uzupełnienia.

```javascript
// Przykład: komentarz precyzyjnie opisujący intencję = lepsza sugestia
// Pobiera użytkownika z MongoDB po ID, rzuca NotFoundError jeśli nie istnieje
async function getUserById(id) {
  // Copilot zasugeruje kod zgodny z powyższym opisem
}
```

### 2. Copilot Chat – kontekst, uczestnicy i komendy ukośnikowe

Copilot Chat to panel rozmowy umożliwiający zadawanie pytań i wydawanie poleceń dotyczących kodu. Kluczem do efektywnego korzystania z czatu jest rozumienie **uczestników** (`@`) i **komend ukośnikowych** (`/`).

**Uczestnicy kontekstowi (`@`):**

- **`@workspace`** – przeszukuje cały projekt i dołącza relevantne pliki do zapytania. Niezbędny przy pytaniach architekturalnych (`@workspace jak jest zorganizowana warstwa serwisów?`).
- **`@vscode`** – odpowiada na pytania dotyczące samego edytora, ustawień i skrótów.
- **`@terminal`** – analizuje ostatnie polecenie i błąd w terminalu, proponuje poprawkę.
- **`@github`** – (Copilot Enterprise) dostęp do repozytoriów, PR i Issues na GitHub.

**Komendy ukośnikowe (`/`):**

| Komenda | Opis |
|---|---|
| `/explain` | Wyjaśnia zaznaczony kod lub cały plik |
| `/fix` | Naprawia błąd (najlepiej z zaznaczonym fragmentem) |
| `/tests` | Generuje testy jednostkowe dla zaznaczonej funkcji lub pliku |
| `/doc` | Dodaje dokumentację JSDoc/TSDoc do zaznaczonego kodu |
| `/simplify` | Upraszcza zaznaczony fragment |
| `/new` | Tworzy nowy plik lub projekt na podstawie opisu |

**Przykład użycia `/fix` z `@terminal`:**

```bash
# W terminalu pojawia się błąd:
Error: Cannot find module './services/auth.service'
    at Function.Module._resolveFilename (node:internal/modules/cjs/loader:1048:15)
```

Zamiast ręcznie szukać przyczyny, w Copilot Chat: `@terminal /fix` → Copilot analizuje błąd i wskazuje brakujący plik lub błędną ścieżkę importu.

### 3. Generowanie testów jednostkowych z `/tests`

Automatyczne generowanie testów to jedna z najbardziej wartościowych funkcji Copilota w codziennej pracy zespołowej. Komenda `/tests` analizuje sygnaturę i implementację funkcji, identyfikuje ścieżki wykonania (happy path, edge cases, błędy) i generuje zestaw testów gotowych do uruchomienia.

**Workflow generowania testów:**

1. Zaznacz funkcję lub cały plik (`Ctrl+A` dla całego pliku).
2. W Copilot Chat napisz `/tests` (opcjonalnie dodaj instrukcje: `/tests używaj Jest, mockuj moduł mongoose`).
3. Przejrzyj wygenerowane testy – sprawdź czy pokrywają kluczowe przypadki.
4. Kliknij *Insert into new file* lub skopiuj do istniejącego pliku testowego.
5. Uruchom testy i popraw ewentualne importy lub mocki.

**Przykład – generowanie testów dla serwisu Node.js:**

```javascript
// Plik: src/services/user.service.js
const User = require('../models/user.model');

async function getUserById(id) {
  const user = await User.findById(id);
  if (!user) throw new Error(`User ${id} not found`);
  return user;
}

module.exports = { getUserById };
```

Po zastosowaniu `/tests` Copilot wygeneruje m.in.:

```javascript
// Plik: src/services/user.service.test.js
const { getUserById } = require('./user.service');
const User = require('../models/user.model');

jest.mock('../models/user.model');

describe('getUserById', () => {
  it('zwraca użytkownika gdy istnieje', async () => {
    const mockUser = { _id: '123', name: 'Jan Kowalski' };
    User.findById.mockResolvedValue(mockUser);

    const result = await getUserById('123');

    expect(result).toEqual(mockUser);
    expect(User.findById).toHaveBeenCalledWith('123');
  });

  it('rzuca błąd gdy użytkownik nie istnieje', async () => {
    User.findById.mockResolvedValue(null);

    await expect(getUserById('999')).rejects.toThrow('User 999 not found');
  });
});
```

**Wskazówki jakości:**

- Dodaj do zapytania informację o frameworku testowym, żeby uniknąć mieszania składni Mocha i Jest.
- Jeśli Copilot nie mockuje zewnętrznych zależności, doprecyzuj: `/tests zamockuj wszystkie połączenia z bazą danych`.
- Zawsze weryfikuj czy testy faktycznie testują logikę biznesową, a nie tylko wywołania metod.

### 4. Refaktoryzacja wspomagana przez Copilot

Copilot Chat jest szczególnie przydatny przy refaktoryzacji, ponieważ potrafi uwzględnić kontekst całego pliku lub modułu i zaproponować zmiany zgodne z istniejącym stylem kodu.

**Typowe scenariusze refaktoryzacji:**

- **Ekstrakcja funkcji** – zaznacz fragment kodu i napisz `/simplify` lub `wyodrębnij tę logikę do osobnej funkcji o nazwie validatePayload`.
- **Migracja składni** – `przekonwertuj wszystkie .then().catch() na async/await`.
- **Zwiększenie czytelności** – `zmień warunki zagnieżdżone na early return`.
- **Poprawa typowania TypeScript** – `dodaj ścisłe typy do parametrów i wartości zwracanych`.

**Przykład refaktoryzacji Express route handlera:**

```javascript
// Przed – kod trudny do testowania, logika wymieszana z HTTP
app.post('/users', async (req, res) => {
  try {
    if (!req.body.email || !req.body.password) {
      return res.status(400).json({ error: 'Missing fields' });
    }
    const existing = await User.findOne({ email: req.body.email });
    if (existing) return res.status(409).json({ error: 'Email taken' });
    const user = new User(req.body);
    await user.save();
    res.status(201).json(user);
  } catch (e) {
    res.status(500).json({ error: e.message });
  }
});
```

Prompt do Copilota: *„Wyodrębnij logikę biznesową do funkcji `createUser` w osobnym serwisie. Handler powinien tylko wywoływać serwis i mapować wyniki na odpowiedzi HTTP."*

Po refaktoryzacji logika biznesowa trafia do `user.service.js`, staje się testowalną jednostką, a sam handler jest cienką warstwą HTTP.

**Refaktoryzacja plikowa z `@workspace`:**

Dla większych zmian obejmujących wiele plików warto użyć: `@workspace znajdź wszystkie miejsca gdzie używamy callback-based API mongoose i zaproponuj migrację na async/await`. Copilot wskazuje pliki i pokazuje przykłady zmian, które możesz naśladować.

### 5. Instrukcje własne – plik `.github/copilot-instructions.md`

GitHub Copilot obsługuje plik **`.github/copilot-instructions.md`**, który działa jako stały system prompt dla całego repozytorium – odpowiednik `.cursorrules` z poprzedniej sekcji. Instrukcje z tego pliku są automatycznie dołączane do każdego zapytania w Copilot Chat.

**Struktura przykładowych instrukcji:**

```markdown
# Instrukcje dla GitHub Copilot

## Projekt
Aplikacja SaaS do zarządzania projektami budowlanymi.
Backend: Node.js 20 + Express 4 + MongoDB 7 (Mongoose 8).
Frontend: Angular 17 (standalone components, Angular Signals).

## Standardy kodu
- Typescript strict mode – zawsze typuj zwracane wartości funkcji
- Async/await zamiast Promise chains
- Nazwy zmiennych i funkcji: camelCase (angielskie)
- Nazwy klas i interfejsów: PascalCase
- Pliki: kebab-case.ts

## Testy
- Framework: Jest (backend), Karma + Jasmine (frontend)
- Mockuj zewnętrzne zależności (mongoose, http)
- Każda funkcja publiczna powinna mieć co najmniej 2 testy

## Czego unikać
- Nie używaj `any` w TypeScript bez uzasadnienia
- Nie commituj sekretów ani zmiennych środowiskowych
```

Plik jest wersjonowany razem z projektem, co oznacza że cały zespół korzysta z tych samych instrukcji. To gwarantuje spójność generowanego kodu niezależnie od tego, kto prowadzi sesję z Copilotem.

### 6. Copilot w pracy zespołowej – GitHub Pull Requests i Code Review

Rozszerzenie **GitHub Pull Requests and Issues** w połączeniu z Copilotem umożliwia asystowanie przy przeglądach kodu bezpośrednio w VS Code. Copilot może:

- **Wygenerować opis PR** na podstawie dołączonych commitów (`@github /pr` w czacie lub przycisk w widoku PR),
- **Wyjaśnić diff** – zaznacz zmienione linie i zapytaj `wyjaśnij tę zmianę i jej wpływ na resztę systemu`,
- **Zasugerować ulepszenia** – `@workspace sprawdź ten diff pod kątem potencjalnych problemów z wydajnością lub bezpieczeństwem`.

**Przykład workflow code review z Copilotem:**

```bash
# 1. Pobierz gałąź PR lokalnie
git fetch origin feature/payment-integration
git checkout feature/payment-integration

# 2. Otwórz diff w VS Code
code --diff main feature/payment-integration

# 3. Zaznacz fragment z potencjalnym błędem i użyj Copilot Chat:
# "Czy ta funkcja jest podatna na race condition przy równoległych żądaniach?"
```

Copilot analizuje zaznaczony kod w kontekście całego PR i odpowiada na pytania dotyczące poprawności, wydajności i bezpieczeństwa – przyspieszając proces przeglądu bez zastępowania osądu ludzkiego recenzenta.

## Zadania praktyczne

### Zadanie 1 — Łatwe: Konfiguracja Copilota i eksploracja projektu

**Temat:** Zainstaluj rozszerzenie GitHub Copilot w VS Code i skonfiguruj je dla istniejącego projektu Node.js + Express. Utwórz plik `.github/copilot-instructions.md` definiujący stack technologiczny, konwencje nazewnicze i preferowany styl testów. Następnie, używając Copilot Chat z uczestnikiem `@workspace`, zadaj co najmniej trzy pytania eksploracyjne dotyczące struktury projektu i porównaj trafność odpowiedzi z zapytaniami bez `@workspace`.

---

### Zadanie 2 — Średnie: Generowanie i weryfikacja testów jednostkowych

**Temat:** Wybierz istniejący moduł serwisowy w projekcie Node.js (min. 3 funkcje z logiką biznesową) i wygeneruj dla niego kompletny zestaw testów jednostkowych używając komendy `/tests` w Copilot Chat. Upewnij się, że wygenerowane testy: mockują połączenie z MongoDB, pokrywają happy path i co najmniej jeden przypadek błędu dla każdej funkcji, są uruchamialne przez `npx jest`. Uruchom testy, napraw ewentualne błędy importów lub mocków i doprowadź do zielonego wyniku.

---

### Zadanie 3 — Średnie: Refaktoryzacja tras Express z Inline Chat

**Temat:** Weź plik routera Express zawierający co najmniej 3 handlery z logiką biznesową wymieszaną z kodem HTTP (walidacja, zapisy do bazy, obsługa błędów wewnątrz handlera). Używając Copilot Inline Chat (`Ctrl+I`), przeprowadź refaktoryzację: wyodrębnij logikę do osobnych funkcji serwisowych, dodaj obsługę błędów przez middleware i dodaj typy TypeScript do parametrów. Po każdej zmianie uruchom istniejące testy, aby upewnić się, że refaktoryzacja nie zepsuła zachowania. Udokumentuj w komentarzu każdą przeprowadzoną zmianę.

---

### Zadanie 4 — Zaawansowane: Automatyzacja code review w pipeline GitHubie

**Temat:** Stwórz workflow GitHub Actions (plik `.github/workflows/copilot-review.yml`), który przy każdym otwarciu Pull Requesta uruchamia skrypt Node.js korzystający z GitHub API i OpenAI API (lub Anthropic API). Skrypt powinien: pobrać diff PR, wysłać go do modelu z prompem w stylu code review (sprawdzenie błędów, bezpieczeństwa, pokrycia testami), opublikować wygenerowany komentarz jako review na PR przez GitHub API. Zabezpiecz klucze API jako GitHub Secrets. Przetestuj workflow na testowym PR zawierającym co najmniej jeden celowy błąd.
