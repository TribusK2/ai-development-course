# 4.2 Angular dla AI Developera: Architektura komponentów i szybka integracja z backendem Node.js

**Moduł 4: Szybkie Wdrażanie (Ship Fast) w oparciu o Angular**

---

## Teoria

### 1. Architektura Angular w kontekście aplikacji AI

Angular to **opiniowany framework** — narzuca określoną strukturę projektu, co jest zaletą przy pracy zespołowej i skalowaniu. Dla AI developera kluczowe jest szybkie opanowanie kilku filarów tej architektury, by nie tracić czasu na konfigurację, a skupić się na funkcjonalnościach AI.

Fundamentalne elementy Angulara:

| Element | Rola w aplikacji AI |
|---------|---------------------|
| **Component** | Jednostka UI — okno czatu, karta wyniku, panel ustawień |
| **Service** | Logika biznesowa i komunikacja z API (OpenAI, własny backend) |
| **HttpClient** | Wysyłanie zapytań HTTP do Node.js / API zewnętrznych |
| **Router** | Nawigacja między widokami (czat, historia, ustawienia) |
| **Signal / RxJS** | Reaktywne zarządzanie stanem (odpowiedź AI w czasie rzeczywistym) |

W nowoczesnym Angularze (v17+) dominują **standalone components** — komponenty bez przynależności do NgModule, importujące zależności bezpośrednio. Zmniejsza to boilerplate i przyspiesza prototypowanie.

---

### 2. Wzorzec Smart/Dumb Components w aplikacjach AI

Separacja odpowiedzialności to klucz do utrzymywalnego kodu. W aplikacjach AI naturalnie pojawia się wzorzec **Smart / Dumb** (znany też jako Container / Presentational):

- **Smart component** (kontener): odpytuje serwis, zarządza stanem, obsługuje błędy. Nie wie, jak renderować dane.
- **Dumb component** (prezentacyjny): przyjmuje dane przez `@Input()`, emituje zdarzenia przez `@Output()`. Nie wie, skąd pochodzą dane.

```typescript
// smart: chat-container.component.ts
@Component({
  selector: 'app-chat-container',
  standalone: true,
  imports: [ChatMessagesComponent, ChatInputComponent],
  template: `
    <app-chat-messages [messages]="messages()" />
    <app-chat-input (send)="onSend($event)" [loading]="loading()" />
  `
})
export class ChatContainerComponent {
  private chatService = inject(ChatService);
  messages = this.chatService.messages;   // Signal<Message[]>
  loading = this.chatService.loading;     // Signal<boolean>

  onSend(text: string) {
    this.chatService.sendMessage(text);
  }
}
```

```typescript
// dumb: chat-input.component.ts
@Component({
  selector: 'app-chat-input',
  standalone: true,
  imports: [MatFormFieldModule, MatInputModule, MatButtonModule, FormsModule],
  template: `
    <mat-form-field appearance="outline">
      <textarea matInput [(ngModel)]="text" placeholder="Pytanie..."></textarea>
    </mat-form-field>
    <button mat-raised-button color="primary"
            [disabled]="loading || !text.trim()"
            (click)="send.emit(text); text = ''">
      Wyślij
    </button>
  `
})
export class ChatInputComponent {
  @Input() loading = false;
  @Output() send = new EventEmitter<string>();
  text = '';
}
```

### 3. Serwisy Angular i komunikacja z backendem Node.js

Serwis (`@Injectable`) to singleton zarządzający logiką i stanem. W aplikacji AI typowy serwis odpowiada za komunikację z backendem Node.js — wysyłanie zapytań i zarządzanie odpowiedziami.

**Backend Node.js (Express) — prosty endpoint AI:**

```javascript
// server.js
import express from 'express';
import OpenAI from 'openai';

const app = express();
app.use(express.json());

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

app.post('/api/chat', async (req, res) => {
  const { message, history } = req.body;

  const completion = await openai.chat.completions.create({
    model: 'gpt-4o',
    messages: [
      { role: 'system', content: 'Jesteś pomocnym asystentem.' },
      ...history,
      { role: 'user', content: message },
    ],
  });

  res.json({ reply: completion.choices[0].message.content });
});

app.listen(3000, () => console.log('Backend działa na porcie 3000'));
```

**Serwis Angular wywołujący endpoint:**

```typescript
// chat.service.ts
import { Injectable, signal, inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';

export interface Message {
  role: 'user' | 'assistant';
  content: string;
}

@Injectable({ providedIn: 'root' })
export class ChatService {
  private http = inject(HttpClient);
  private apiUrl = '/api/chat';

  messages = signal<Message[]>([]);
  loading = signal(false);

  sendMessage(text: string) {
    const userMsg: Message = { role: 'user', content: text };
    this.messages.update(msgs => [...msgs, userMsg]);
    this.loading.set(true);

    this.http.post<{ reply: string }>(this.apiUrl, {
      message: text,
      history: this.messages().slice(0, -1),  // historia bez ostatniego
    }).subscribe({
      next: ({ reply }) => {
        this.messages.update(msgs => [...msgs, { role: 'assistant', content: reply }]);
        this.loading.set(false);
      },
      error: () => this.loading.set(false),
    });
  }
}
```

### 4. Proxy deweloperski — Angular CLI i CORS

Podczas developmentu Angular działa na `localhost:4200`, a backend Node.js na `localhost:3000`. Przeglądarki blokują takie żądania cross-origin (CORS). Zamiast konfigurować CORS na backendzie w środowisku deweloperskim, Angular CLI oferuje wbudowany **proxy**:

```json
// proxy.conf.json
{
  "/api": {
    "target": "http://localhost:3000",
    "secure": false,
    "changeOrigin": true
  }
}
```

```json
// angular.json — fragment konfiguracji serve
"serve": {
  "options": {
    "proxyConfig": "proxy.conf.json"
  }
}
```

```bash
ng serve   # proxy aktywny automatycznie
```

Żądania do `/api/*` z Angulara są transparentnie przekierowywane do Node.js. W produkcji (AWS Lambda + CloudFront) analogiczną rolę pełni konfiguracja origins API Gateway lub CloudFront behaviors.

### 5. Reaktywność z Signals i obsługa strumieniowania odpowiedzi AI

Angular Signals (stabilne od v17) to prostszy model reaktywności niż RxJS dla podstawowych przypadków: aktualizacja sygnału automatycznie odświeża powiązane widoki.

Dla **strumieniowania** odpowiedzi AI (Server-Sent Events lub chunked response) wygodniej użyć `EventSource` lub `fetch` z `ReadableStream`:

```typescript
// Strumieniowanie przez Fetch API w serwisie Angular
async streamMessage(text: string) {
  this.loading.set(true);
  const assistantMsg: Message = { role: 'assistant', content: '' };
  this.messages.update(msgs => [...msgs, { role: 'user', content: text }, assistantMsg]);

  const response = await fetch('/api/chat/stream', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ message: text }),
  });

  const reader = response.body!.getReader();
  const decoder = new TextDecoder();

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    const chunk = decoder.decode(value, { stream: true });
    this.messages.update(msgs => {
      const last = { ...msgs[msgs.length - 1] };
      last.content += chunk;
      return [...msgs.slice(0, -1), last];
    });
  }
  this.loading.set(false);
}
```

Widok automatycznie aktualizuje się z każdym fragmentem — użytkownik widzi odpowiedź AI pojawiającą się słowo po słowie, jak w ChatGPT.

### 6. Konfiguracja proxy i zmienne środowiskowe

Angular CLI obsługuje pliki środowiskowe (`environments/`), które pozwalają definiować różne wartości (URL backendu, klucze API) dla środowiska deweloperskiego i produkcyjnego:

```typescript
// environments/environment.ts (dev)
export const environment = {
  production: false,
  apiUrl: '/api',   // proxy deweloperski
};

// environments/environment.prod.ts (prod)
export const environment = {
  production: true,
  apiUrl: 'https://xyz.execute-api.eu-west-1.amazonaws.com/prod',  // AWS API Gateway
};
```

```bash
ng build                          # używa environment.ts
ng build --configuration production  # używa environment.prod.ts
```

Dzięki temu ten sam kod serwisu Angular działa zarówno lokalnie (z proxy na Node.js w Dockerze), jak i w produkcji (z AWS Lambda przez API Gateway) — bez żadnych zmian w kodzie aplikacji.

## Zadania praktyczne

### Zadanie 1 — Łatwe
**Prosty serwis komunikacji z backendem**

Utwórz serwis Angular `AiService` z metodą `ask(question: string): Observable<string>`, która wysyła `POST /api/ask` do prostego backendu Node.js/Express zwracającego losową odpowiedź tekstową (bez prawdziwego AI). Wstrzyknij serwis do komponentu i wyświetl odpowiedź w szablonie po kliknięciu przycisku. Skonfiguruj proxy deweloperski Angular CLI, by uniknąć błędów CORS.

---

### Zadanie 2 — Średnie
**Wzorzec Smart/Dumb dla widoku konwersacji**

Zaprojektuj i zaimplementuj dwa standalone komponenty:
- `ChatContainerComponent` (smart) — zarządza historią wiadomości (`Signal<Message[]>`) i wywołuje `ChatService`,
- `MessageListComponent` (dumb) — wyświetla listę wiadomości przez `@Input()`, stylizując wiadomości użytkownika i asystenta inaczej (np. `mat-card` z różnym `color`).

Połącz je w routingu pod ścieżką `/chat`. Przetestuj całość uruchamiając backend Node.js lokalnie w Dockerze.

---

### Zadanie 3 — Średnie
**Konfiguracja środowisk i integracja z prawdziwym API**

Skonfiguruj pliki `environment.ts` i `environment.prod.ts` tak, by `apiUrl` wskazywał odpowiednio na lokalny backend (port 3000) i URL AWS API Gateway. Zmodyfikuj `ChatService`, by korzystał z `environment.apiUrl`. Podłącz backend Node.js do OpenAI API (endpoint `POST /api/chat`) i zweryfikuj działanie całego przepływu: Angular → Node.js → OpenAI → Angular. Uruchom środowisko deweloperskie przez `docker compose up`.

---

### Zadanie 4 — Zaawansowane
**Strumieniowanie odpowiedzi AI z animowanym widokiem**

Rozbuduj backend Node.js o endpoint `POST /api/chat/stream`, który strumieniuje odpowiedź OpenAI chunk po chunku (OpenAI streaming API z `stream: true`). Po stronie Angular zaimplementuj metodę `streamMessage()` używającą Fetch API z `ReadableStream`. Wyświetlaj przyrastającą odpowiedź w komponencie Angular Material (`mat-card` z efektem kursora migającego na końcu tekstu, zrealizowanym przez CSS animation). Całość uruchom w środowisku Docker Compose i zweryfikuj płynność animacji strumieniowania.
