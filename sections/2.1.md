# 2.1 Konfiguracja edytora Cursor: Indeksowanie kodu, funkcja Composer i praca w trybie terminala AI

> **Moduł 2: Ekosystem Vibe Coding i AI-first IDE**

---

## Teoria

### 1. Cursor – czym jest AI-first IDE i dlaczego zmienia zasady gry?

Cursor to edytor kodu zbudowany jako fork VS Code, przeprojektowany od podstaw z myślą o integracji z modelami językowymi. W odróżnieniu od wtyczek (np. GitHub Copilot) Cursor nie jest dodatkiem do istniejącego edytora – model AI jest tu obywatelem pierwszej klasy: ma dostęp do całego projektu, może modyfikować wiele plików jednocześnie, wykonywać polecenia w terminalu i iteracyjnie poprawiać własne odpowiedzi.

**Kluczowe pojęcia:**

- **AI-first IDE** – edytor, w którym interfejs, skróty klawiszowe i architektura są zoptymalizowane wokół współpracy z modelem językowym, a nie tylko go hostują.
- **Underlying model** – Cursor domyślnie korzysta z Claude 3.5/3.7 Sonnet lub GPT-4o (konfigurowalne w ustawieniach). Wybór modelu wpływa na szybkość, koszt i jakość odpowiedzi.
- **Context window** – podobnie jak przy pracy z API, Cursor wysyła do modelu fragmenty kodu jako kontekst. Świadome zarządzanie tym, co trafia do okna kontekstowego, jest kluczem do precyzyjnych wyników.

Cursor jest szczególnie efektywny przy pracy z dużymi bazami kodu, gdzie ręczne wyszukiwanie i modyfikacja plików byłyby czasochłonne. Jego największą przewagą jest zdolność do *rozumienia projektu jako całości* – nie tylko edytowanego pliku.

---

### 2. Indeksowanie kodu – jak Cursor „czyta" Twój projekt

Po otwarciu folderu Cursor tworzy lokalny indeks projektu. To proces, który przekształca kod źródłowy w reprezentacje wektorowe (embeddingi), umożliwiając semantyczne wyszukiwanie po całej bazie kodu. Dzięki temu możesz zadać pytanie w naturalnym języku, np. *„gdzie jest obsługiwany błąd autoryzacji?"*, a Cursor wskaże odpowiednie pliki i linie.

**Jak działa indeksowanie:**

1. Cursor skanuje pliki projektu (respektując `.gitignore` i ustawienia w `cursorignore`).
2. Każdy plik jest dzielony na chunki i zamieniany na wektory (embeddingi).
3. Indeks jest przechowywany lokalnie – nie jest wysyłany na serwery Cursor bez Twojej zgody (tryb Privacy Mode).
4. Przy pytaniu (w trybie Chat) Cursor wykonuje similarity search po indeksie i dołącza najtrafniejsze fragmenty do zapytania do modelu.

**Praktyczne wskazówki:**

- Plik **`.cursorignore`** działa jak `.gitignore` – wyklucza foldery `node_modules`, `dist`, pliki logów i inne zasoby, które nie powinny trafić do indeksu. Dobra konfiguracja tego pliku znacząco przyspiesza indeksowanie i poprawia jakość odpowiedzi.
- Możesz ręcznie odświeżyć indeks w Command Palette (`Ctrl+Shift+P` → *Cursor: Reindex Codebase*).
- Im lepiej opisany jest kod (nazwy funkcji, komentarze), tym trafniej Cursor lokalizuje odpowiednie fragmenty.

```bash
# Przykład .cursorignore dla projektu Node.js + Angular
node_modules/
dist/
.angular/
coverage/
*.log
.env
```

### 3. Tryb Chat i Inline Edit – podstawy interakcji z modelem

Cursor oferuje dwa główne tryby pracy z AI w codziennym kodowaniu:

**Chat (`Ctrl+L`)** otwiera panel boczny, w którym możesz prowadzić rozmowę z modelem w kontekście całego projektu. Możesz:
- zadawać pytania o architekturę (`@codebase` – przeszuka indeks),
- dołączać konkretne pliki do kontekstu (`@filename`),
- prosić o wygenerowanie kodu, który następnie możesz zaakceptować jednym kliknięciem.

**Inline Edit (`Ctrl+K`)** uruchamia mini-okno bezpośrednio w edytorze – zaznaczasz fragment kodu, opisujesz zmianę, a model pokazuje diff do zaakceptowania lub odrzucenia. To najszybszy sposób na lokalne modyfikacje (refaktoryzacje, korekty nazw, poprawki logiki).

**Przykład użycia Inline Edit:**

```javascript
// Przed: nieczytelna funkcja sprawdzająca uprawnienia
function chk(u, r) { return u.roles && u.roles.includes(r); }

// Po napisaniu Ctrl+K: "rename parameters, add JSDoc, return early if no roles"
/**
 * Sprawdza, czy użytkownik posiada wskazaną rolę.
 * @param {Object} user - Obiekt użytkownika
 * @param {string} role - Nazwa roli do sprawdzenia
 * @returns {boolean}
 */
function hasRole(user, role) {
  if (!user.roles) return false;
  return user.roles.includes(role);
}
```

Różnica między Chat a Inline Edit: Chat służy do eksploracji i planowania, Inline Edit – do precyzyjnych, lokalnych zmian.

### 4. Composer – praca z wieloma plikami jednocześnie

Composer (`Ctrl+I` lub `Ctrl+Shift+I` dla trybu pełnoekranowego) to najbardziej zaawansowana funkcja Cursora, wyróżniająca go spośród konkurencji. Umożliwia **edycję wielu plików w ramach jednego zapytania** – model może samodzielnie stworzyć nowe pliki, zmodyfikować istniejące i wyświetlić kompletny diff do zatwierdzenia.

**Kiedy używać Composera:**

- Dodawanie nowej funkcjonalności obejmującej kilka warstw (np. nowy endpoint + serwis + moduł Angular),
- Refaktoryzacja całego modułu (np. migracja z `Promise` na `async/await`),
- Generowanie scaffoldingu (struktura katalogów, boilerplate dla nowego mikroserwisu),
- Jednoczesna aktualizacja testów jednostkowych i implementacji.

**Przykład prompta do Composera:**

> Stwórz endpoint `POST /api/users/:id/avatar` w Express.js. Endpoint powinien:
> - przyjmować plik multipart/form-data,
> - walidować typ MIME (tylko `image/jpeg`, `image/png`),
> - zapisywać plik do folderu `uploads/avatars/`,
> - zwracać ścieżkę do pliku w odpowiedzi JSON.
> Zaktualizuj też plik routera i dodaj test jednostkowy w Jest.

Composer wyświetla listę wszystkich planowanych zmian (added/modified/deleted files) zanim cokolwiek zostanie zapisane. Możesz przeglądać każdy plik po kolei i zaakceptować lub odrzucić poszczególne zmiany.

**Tryb Agent w Composerze** (opcja *Agent mode*) pozwala modelowi na wykonywanie poleceń w terminalu i iteracyjne poprawianie kodu na podstawie błędów kompilacji lub testów – bez Twojej interwencji między krokami.

### 5. Praca w trybie terminala AI

Cursor integruje AI z wbudowanym terminalem. Kluczowa funkcja: **`Ctrl+K` w terminalu** – zamiast wpisywać polecenie ręcznie, opisujesz cel w języku naturalnym, a Cursor generuje komendę do zatwierdzenia.

**Przykłady użycia:**

```bash
# Opis: "znajdź wszystkie pliki TypeScript zmodyfikowane w ciągu ostatnich 7 dni"
# Cursor generuje:
find . -name "*.ts" -newer $(date -d '7 days ago' +%Y-%m-%d) -not -path "*/node_modules/*"

# Opis: "uruchom testy tylko dla modułu 'auth', z flagą verbose"
# Cursor generuje:
npx jest --testPathPattern="auth" --verbose
```

**Terminal Debug** – gdy w terminalu pojawia się błąd, możesz kliknąć *Debug with AI* (lub użyć `Ctrl+Shift+R`), aby Cursor przeanalizował traceback i zaproponował poprawkę bezpośrednio w kodzie.

Tryb terminala AI jest szczególnie wartościowy przy pracy z DevOps: pisanie skryptów Bash, konfiguracja pipeline'ów, zarządzanie repozytoriami Git za pomocą opisów zamiast zapamiętywania składni poleceń.

### 6. Konfiguracja reguł projektu – plik `.cursorrules`

`.cursorrules` (lub nowy format: folder `.cursor/rules/`) to mechanizm dostarczania **stałego kontekstu i instrukcji** do każdego zapytania w ramach projektu. Działa jako system prompt na poziomie projektu.

**Co warto umieszczać w `.cursorrules`:**

- Stack technologiczny i konwencje (`"Używaj Angular 17 ze standalone components, RxJS, NgRx"`),
- Styl kodu (`"Zawsze używaj async/await zamiast .then()"`),
- Kontekst domenowy (`"To jest aplikacja B2B SaaS do zarządzania flotą pojazdów"`),
- Często pomijane reguły (`"Testy jednostkowe: pokrycie min. 80%, używaj Testing Library"`).

```markdown
# .cursorrules

## Stack
- Backend: Node.js 20 + Express + MongoDB (Mongoose)
- Frontend: Angular 17 (standalone components, signals)
- Testy: Jest (backend), Karma + Jasmine (frontend)
- Styl: ESLint + Prettier, brak średników, pojedyncze cudzysłowy

## Konwencje
- Nazwy plików: kebab-case
- Klasy i interfejsy: PascalCase
- Zmienne i funkcje: camelCase
- Zawsze typuj odpowiedzi API (interface + DTO)

## Ważne
- Nie używaj `var`, tylko `const` i `let`
- Asynchroniczność: wyłącznie async/await
- Obsługa błędów: zawsze try/catch + logowanie
```

Dobrze skonfigurowany `.cursorrules` redukuje konieczność powtarzania tych samych instrukcji w każdym prompcie i znacząco poprawia spójność generowanego kodu z resztą projektu.

### 7. Efektywny workflow z Cursor – najlepsze praktyki

Znajomość poszczególnych funkcji Cursora to punkt startowy. Kluczem do efektywnej pracy jest wypracowanie **spójnego workflow**, który minimalizuje kontekst-switching i maksymalizuje precyzję modelu.

**Rekomendowany workflow dla nowej funkcjonalności:**

1. **Planowanie w Chat** (`Ctrl+L` z `@codebase`): opisz funkcjonalność i zapytaj o plan implementacji. Na tym etapie identyfikujesz pliki do modyfikacji i potencjalne problemy.
2. **Implementacja w Composerze** (`Ctrl+I`): przekaż plan jako prompt, wskaż pliki kontekstu (`@plik1 @plik2`). Composer generuje diff.
3. **Przegląd i akceptacja**: przejrzyj każdy plik, zaakceptuj/odrzuć zmiany. Nie akceptuj wszystkiego na ślepo.
4. **Korekty Inline** (`Ctrl+K`): dla drobnych poprawek po przeglądzie.
5. **Testy w terminalu** (`Ctrl+K` w terminalu): uruchom testy i jeśli wystąpią błędy, użyj *Debug with AI*.

**Typowe pułapki:**

- **Zbyt duże prompty** – Composer traci fokus przy zbyt wielu wymaganiach naraz. Dziel zadania na etapy.
- **Brak kontekstu** – bez `@pliku` lub `@codebase` model generuje kod „w próżni", niekompatybilny z projektem.
- **Ślepa akceptacja diffów** – zawsze czytaj wygenerowany kod. Modele popełniają błędy logiczne, których nie wychwytuje kompilator.

## Zadania praktyczne

### Zadanie 1 — Łatwe: Konfiguracja Cursora dla projektu Node.js

**Temat:** Skonfiguruj edytor Cursor dla istniejącego projektu backendowego Node.js + Express + MongoDB. Utwórz plik `.cursorignore` wykluczający zbędne foldery oraz plik `.cursorrules` zawierający informacje o stosowanym stacku, konwencjach nazewniczych i preferowanym stylu kodu. Zweryfikuj działanie indeksowania, zadając pytanie o strukturę projektu w trybie Chat z użyciem `@codebase`.

---

### Zadanie 2 — Średnie: Generowanie modułu Angular przy użyciu Composera

**Temat:** Wykorzystaj Composera w edytorze Cursor, aby wygenerować kompletny moduł Angular obsługujący listę zasobów (np. listę użytkowników). Moduł powinien zawierać: komponent listy (standalone), serwis komunikujący się z REST API oraz model TypeScript. W prompcie do Composera odwołaj się do istniejącego pliku serwisu HTTP w projekcie jako wzorca. Przejrzyj wygenerowany diff i wprowadź co najmniej jedną ręczną korektę przed akceptacją.

---

### Zadanie 3 — Średnie: Automatyzacja powtarzalnego zadania skryptem Bash przez terminal AI

**Temat:** Używając trybu terminala AI (`Ctrl+K` w terminalu Cursora), wygeneruj i wykonaj skrypt Bash, który: przeszuka projekt w poszukiwaniu plików `.log` starszych niż 7 dni, wyświetli ich nazwy i rozmiary, a następnie przeniesie je do folderu archiwum (`logs/archive/`). Następnie poproś Cursora o zmodyfikowanie skryptu tak, aby przyjmował liczbę dni jako argument wejściowy. Zapisz gotowy skrypt do pliku `scripts/cleanup-logs.sh`.

---

### Zadanie 4 — Zaawansowane: Iteracyjna implementacja funkcjonalności z użyciem trybu Agent

**Temat:** Włącz **tryb Agent** w Composerze Cursora i poproś go o zaimplementowanie od zera funkcjonalności resetowania hasła w aplikacji Express + MongoDB: endpoint `POST /api/auth/reset-password/request` (generowanie tokenu i symulacja wysyłki e-mail przez `console.log`), endpoint `POST /api/auth/reset-password/confirm` (weryfikacja tokenu, zmiana hasła), model Mongoose dla tokenu resetującego, testy jednostkowe w Jest dla obu endpointów. Pozwól agentowi iteracyjnie uruchamiać testy i poprawiać kod aż do uzyskania zielonego wyniku. Udokumentuj, ile iteracji wymagał agent i jakie błędy napotykał.
